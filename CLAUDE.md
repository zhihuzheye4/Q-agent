# 记忆系统规则

本项目专属记忆体系，所有记忆文件遵循三种权限等级。

## 一、权限三原则

| 等级 | AI 写 | AI 读 | AI 改/删 | 用途 |
|------|------|------|---------|------|
| **L1 可写不可读** | 只追加（append-only） | ✗ | ✗ | 历史事实档案，AI 当记录员 |
| **L2 可写可删改不可读** | ✓ | ✗ | ✓ | 人类维护档案，AI 不读 |
| **L3 完全权限** | ✓ | ✓ | ✓ | AI 工作记忆 |

**核心原则**：越重要的记忆越该归 L1。原因：Claude Code 的 Edit 工具有误删风险，L1 纯追加无 Edit 操作，最安全。

## 二、记忆文件清单

### L1 可写不可读（纯追加，AI 不读不改）

| 文件 | 路径 | 用途 | 触发时机 |
|------|------|------|---------|
| 对话记录 | `memory/对话记录.md` | 逐字记录全部对话文本（不含工具调用） | 每轮对话实时追加 |
| 提问回答记录 | `memory/提问回答记录.md` | 触发词区间内的对话逐字记录 | 触发词启停 |
| 核心思想提取 | `memory/核心思想提取.md` | 提取核心思想/脑洞/思路 + AI 逻辑分析 | 紧接对话记录和提问记录存储后生成（不等会话结束） |
| 编年史记忆 | `memory/编年史记忆.md` | 临时记忆覆盖前的完整快照 | 临时记忆覆写前 |
| ADR 架构决策记录 | `memory/ADR架构决策记录.md` | 关键技术选型的背景/选项/取舍 | 每个重大决策追加一条 |
| 思维导图数据源 | `memory/思维导图数据.jsonl` | 思维导图节点事件流 | 功能增删时追加一行 |
| **方案文件夹** | `memory/方案/` | 每次讨论后生成的具体执行方案（AI 不回读） | 讨论达成共识后追加，执行后归档到 `memory/方案/已执行/` |

### L3 完全权限（AI 可读可改）

| 文件 | 路径 | 用途 |
|------|------|------|
| 临时记忆 | `memory/临时记忆.md` | 项目详情 + 上次进度，AI 启动时必读 |
| **项目指导文件** | `memory/项目指导文件.md` | Claude Code 在本项目工作时的导航图 |
| 思维导图 | `memory/思维导图.md` | 脚本渲染的可视化进度条，AI 不直接 Edit |
| 技能工具注册清单 | `memory/技能工具注册清单.md` | 已实现技能/工具/Agent 元数据，编排层查询用 |
| 本项目错误追踪 | `memory/本项目错误追踪.md` | 框架自身 bug（区别于全局工作环境错误） |

### L2 可写可删改不可读

（当前无文件归入此等级，保留以备未来需求）

## 三、关键操作规则

### 启动流程（新窗口打开时）
1. 只读两文件：`G:\agent\CLAUDE.md` + `G:\agent\memory\临时记忆.md`
2. 其余 L1/L2 文件**禁止读取**
3. 读完后第一动作：向用户复述临时记忆中的"下一步"，确认后开工

### 对话记录写入
- 每轮对话实时 append 到 `memory/对话记录.md`
- 只记录用户和 AI 的文本，**不记录工具调用指令**
- 单文件超过 500KB 或 200 轮 → 切片为 `对话记录_YYYY-MM_partNN.md` 封存到 `memory/归档/`

### 提问状态触发词
- 开始：用户输入 `现在开始提问` 或 `【提问】`
- 结束：用户输入 `现在结束提问` 或 `【结束提问】`
- 区间内文本逐字追加到 `memory/提问回答记录.md`

### 核心思想提取
- **触发时机**：紧接对话记录和提问记录存储后生成（不等会话结束）
- 写入 `memory/核心思想提取.md`，两部分：
  1. 核心思想 / 脑洞 / 项目思路
  2. AI 逻辑分析
- 新会话新节追加，不改旧内容

### 临时记忆覆盖（强制流程）
**临时记忆覆写前必须执行**：
1. 将当前临时记忆完整内容追加到 `memory/编年史记忆.md`（L1）
2. 然后才允许覆写 `memory/临时记忆.md`
3. 编年史每节标注时间戳

### 思维导图更新流程
**AI 禁止直接 Edit `memory/思维导图.md`**。操作流程：
1. 向 `memory/思维导图数据.jsonl` 追加一行事件
   - 新增：`{"action":"add","time":"YYYY-MM-DD","name":"功能名","tech":"技术路线","idea":"技术思路"}`
   - 删除：`{"action":"del","time":"YYYY-MM-DD","name":"功能名","reason":"删除原因"}`
2. 调用脚本：`python scripts/update_mindmap.py`
3. 脚本读取 jsonl → 重新渲染 md（双线制：主线存活功能 + 支线已删除功能）

### ADR 决策记录
- 每个重大架构决策追加一条 ADR-NNN
- 新决策取代旧决策时，新条目标注"取代 ADR-XXX"，**旧条目保留不变**
- 纯 append-only

## 四、行为规则

1. **理解核心思想而非字面执行**：用户表述可能用词不当或不准确，必须提取核心思想后按理解执行。此规则优先级最高。
2. **L1 文件禁止 Edit**：只能用 append（追加）方式写入，禁止使用 Edit 工具替换内容。这是防止误删的根本防线。
3. **L3 文件编辑前先 commit**：临时记忆/思维导图/技能注册/错误追踪这些 L3 文件被 Edit 前应先 git commit，靠历史兜底。
4. **临时记忆覆盖前必走编年史**：违反此规则 = 永久丢失进度快照。
5. **思维导图永不直接 Edit**：必须走 jsonl + 脚本路径。

## 五、文件膨胀控制

- 对话记录：> 500KB 或 > 200 轮 → 切片封存到 `memory/归档/`
- 思维导图 jsonl：永远追加，永不切片（单行事件体量小）
- 编年史：永远追加，永不切片（永久档案）
- 其余 L1：永远追加，永不切片

## 六、与全局规则的关系

- 全局工作环境错误档案 `F:\EnglishWenJian\claude-CeShi\工作错误追踪档案\错误追踪总表.md`：记工具调用阻力（网络/环境/依赖）
- 本项目 `memory/本项目错误追踪.md`：记框架功能缺陷（编排/技能/工具调用层）
- 两者不混用

## 七、方案文件夹规则

- 路径：`memory/方案/`（L1 不可读），已执行归档：`memory/方案/已执行/`
- 命名：`方案_YYYY-MM-DD_HHmm_主题简述.md`（归档时改名 `已执行_方案_YYYY-MM-DD_HHmm_主题简述.md`）
- 触发：每次讨论达成共识后，AI 生成具体方案写入此文件夹
- 执行：AI 不回读方案文件，基于当前会话上下文执行
- 归档：执行完毕后移到 `memory/方案/已执行/`

## 八、项目指导文件规则

- 路径：`memory/项目指导文件.md`（L3 完全权限，AI 启动可读）
- 作用：Claude Code 在本项目工作时的导航图——项目方向 + 各模块准备情况 + 下一步可能选项
- 更新时机：每个里程碑完成后更新"当前节点"和"已完成节点"
- "准备好但目前用不上的"方向在文件中标注触发条件，等用户提需求时再调用

# 开发执行规则

## 一、项目定位

Q-agent 是类似 Claude Code 的桌面端 AI 工具，纯软件形式对接本地 LLM，让 LLM 指令被软件理解并执行。

## 二、关键约束

- **永不沙箱**：不引入沙箱模块。LLM 指令通过工具调用层（`q_agent/tools/`）直接执行，仅保留基本输入校验（危险命令黑名单 + 项目根目录保护）
- **本地 LLM 优先**：默认走本地后端，云端按需启用
- **零第三方依赖起步**：CLI 用 argparse，不引入 Typer/Click

## 三、开发哲学：增量盖房

框架阶段一次性做好准备工作（工具链 + 核心骨架 + 指导文件 + 规则）；功能开发阶段增量添加——用户提需求 → 讨论 → 生成方案文件 → 执行 → 更新指导文件。

## 四、编码工具链（第一天配齐）

- ruff（lint + format）
- mypy strict
- pytest + pytest-cov（起步覆盖率 30%，渐进到 70%）
- pre-commit 配置文件备好但不强求安装

commit 前必须：`ruff check . && ruff format --check . && mypy q_agent && pytest -q` 全绿。

## 五、基本安全规则（不是沙箱）

- 所有外部命令通过 `q_agent.tools.executor.execute_shell`，禁 `shell=True`、`os.system`、`subprocess.run(str)`
- `q_agent.tools.safety.check_command` 拦截危险命令黑名单
- `q_agent.tools.safety.check_path` 拦截以 `G:\agent` 为目标的操作
- 测试用 `tmp_path` fixture，禁止以项目根为 cwd

## 六、技能/工具注册约定

- `@skill(name, desc)` 装饰纯函数（技能：面向用户的命令）
- `@tool(name, desc)` 装饰纯函数（工具：LLM 调用的底层操作）
- 函数签名（参数名+类型+返回类型）即协议
- `inspect.signature` 在注册时校验
- 技能元数据进 `memory/技能工具注册清单.md`

## 七、提交规则

- 一个可运行里程碑一个 commit
- commit 前工具链全绿
- commit 信息中文祈使句，首行 ≤50 字，体说明 why

## 八、测试要求

- 核心包覆盖率起步 30%，渐进到 70%
- 每个 skill/tool 必须有 happy-path 测试
- 基本安全必须有拒绝危险命令和受保护路径的用例
- 测试用 `tmp_path` fixture

## 九、文档规则

- 每个包 `__init__.py` 顶部写模块职责说明
- 公开 API 变更同步追加 ADR
- README 仅保快速启动

## 十、本地 LLM 优先

- 默认 `LLMClient` 走本地后端
- 云端后端按需启用且需显式 env 开关
- 不硬编码云端 key

## 十一、C 盘保护

- pip 大型包用 `--target=F:\EnglishWenJian\claude-CeShi\claude-data\pip-packages`
- pip 缓存用 `--cache-dir=F:\EnglishWenJian\claude-CeShi\claude-data\npm-cache`
- HF/模型缓存设 `HF_HOME` / `TRANSFORMERS_CACHE` 到 F 盘

## 十二、项目根目录保护

- 任何 shell/文件操作禁止以 `G:\agent` 为工作目录或删除目标
- 违反即记 `memory/本项目错误追踪.md`

## 十三、错误追踪分线

- 框架自身 bug → `memory/本项目错误追踪.md`（L3，可改）
- 工具调用阻力（网络/环境/依赖）→ 全局 `F:\EnglishWenJian\claude-CeShi\工作错误追踪档案\错误追踪总表.md`
- 两者不混用

## 十四、新功能上线流程

1. 实现
2. 单测
3. `__init__.py` 文档
4. 注册到 `memory/技能工具注册清单.md`
5. 追加 `memory/思维导图数据.jsonl` 一行 → `python scripts/update_mindmap.py`
6. 追加 ADR-NNN 到 `memory/ADR架构决策记录.md`
7. 更新 `memory/项目指导文件.md`（当前节点 + 已完成节点）
8. commit

## 十五、运行期记忆与协作记忆隔离

- `q_agent/memory/` = 框架运行期记忆（Agent 跑起来时用）
- `memory/` 文件层 = 人+AI 协作记忆（开发过程留痕）
- 两者物理隔离，禁止混用

## 十六、方案文件夹规则

- 每次讨论后生成具体方案，存到 `memory/方案/方案_YYYY-MM-DD_HHmm_主题.md`（L1 不可读）
- AI 执行时不回读方案文件，基于当前会话上下文执行
- 执行完毕后归档到 `memory/方案/已执行/`

## 十七、可执行安装包规则

- **每个可运行里程碑必须生成一个 `.exe` 放到 `安装包/` 文件夹**（与 commit 同步）
- 子目录按版本号命名：`安装包/v0.0.1/`、`安装包/v0.0.2/` …
- 打包工具：PyInstaller（dev 依赖，装到 F 盘 venv，不污染 C 盘，不违反"零第三方运行时依赖"原则）
- 打包入口：`q_agent/cli.py` 的 `main()`
- 打包命令（F 盘 venv 内执行）：
  ```bash
  pyinstaller --distpath 安装包 --workpath build --name Q-agent --onefile q_agent/cli.py
  ```
  然后把 `安装包/Q-agent/` 重命名为 `安装包/v{版本}/`
- `.exe` 二进制不入 git（已在 `.gitignore` 忽略），目录结构靠 `.gitkeep` + `README.md` 保留
- 历史版本长期保留，不删旧版本
- 生成后更新 `安装包/README.md` 的版本历史表

## 十八、UI 矢量图资源规则

- **UI 界面调用的所有矢量图存放到 `q_agent/assets/icons/`**
- 该文件夹**只存** UI 界面调用的矢量图（SVG 格式）
- **禁止存放**：位图（PNG/JPG）、字体、音频、配置文件、开发文档图
- 开发文档用的图放 `memory/` 或根目录，与 UI 资源物理隔离
- 命名规范：`图标名-状态.svg`（如 `send-active.svg`、`send-disabled.svg`）
- `q_agent/assets/` 是包内资源，PyInstaller 打包时会一起进 `.exe`
- UI 代码访问方式：`importlib.resources` 或 `pathlib.Path(__file__).parent / "assets"`
- 当前为空目录（含 `.gitkeep`），等 UI 界面实现时再填充

## 十九、指令执行自由度

- 用户下达的指令默认有自由选择空间，非强制按字面执行
- AI 在执行时带判断力：路径/工具/命名等细节可自定，并在执行后说明选择
- 大决策（引入新依赖、改变架构、不可逆操作）仍需先展示方案让用户否决
- 用户用"必须/一定/严格按/不许"等明确措辞 = 强制字面执行，无选择空间

## 二十、分发零外部依赖硬规则

- **最终用户拿到 `.exe` 双击即跑，禁止让用户装任何外部依赖**
- 反例：某些 AI 工具要求用户先在 `requirements.txt` 装 100+ 依赖才能用某功能——此类设计在本项目被禁止
- 所有运行时依赖（Python 解释器 + 第三方库 + 资源文件）必须由 PyInstaller `--onefile` 打包进单个 `.exe`，做到"一个文件交付完整软件"
- 此规则是第十七节"可执行安装包规则"的延伸——不仅每个里程碑打 `.exe`，且 `.exe` 必须自包含
- 开发期依赖（PySide6/PyInstaller/ruff/mypy 等）装在 F 盘 venv，仅供开发者使用，**不进入分发包**
- 依赖选择原则：优先选**能被 PyInstaller 打包**的库；避免引入**动态加载/外部子进程依赖**的库（这类库即便进 .exe 也会在用户机器上要求额外环境）
- .exe 体积是优化目标（避免 1G 软件 + 20G 外部库的"买手机得先买光刻机"荒谬），但**零外部依赖安装**是硬规则，体积可以大但不能让用户装东西
- 引入新运行时依赖前必须验证：能否被 PyInstaller `--onefile` 打包？打包后用户机器零外部依赖即可运行？
