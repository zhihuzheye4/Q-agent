零、调研方式说明

本次调研为单 agent 一次性产出，非多 agent 协作结果

具体做法：派了 1 个 general-purpose agent，让它用 WebSearch / WebFetch 实际查 24 个框架的官方文档和 GitHub 源码，整理成报告归档到 memory/方案/方案_2026-06-29_1430_v0.0.19工具层市场调研.md。整个过程没有多 agent 编排（没做"分头调研 + 交叉审查 + 综合仲裁"）。

单 agent 调研的局限性

  • 视角单一：所有维度由同一个 agent 顺次扫，没有交叉验证，可能漏掉某框架的某关键设计
  • 深度受限：单 agent token 预算有限，每个框架只能查 1-2 个权威源，无法穷尽源码细节
  • 主观偏差：归纳模式和对 Q-agent 的启示由单 agent 一次性给出，没有"反方意见"
  • 耗时集中：本次 agent 跑了约 9.6 分钟（27 次工具调用），全程串行

多 agent 协作调研的备选方案（如需提升可信度可重做）

  • 方案 A：分领域并行 + 综合：4 个 agent 分头查（1 个查工具注册机制 / 1 个查安全策略 / 1 个查 MCP 支持 / 1 个查本地 LLM 适配），最后 1 个综合 agent 仲裁
  • 方案 B：分框架并行 + 综合：24 个框架分给 4-6 个 agent，每个 agent 查 4-6 个框架，最后综合
  • 方案 C：红蓝对抗：1 个 agent 出报告，1 个 agent 批判性审查找漏洞，第 3 个仲裁
  • 方案 D：当前方案 + 复核：保留单 agent 报告，另派 1 个 agent 只做"抽查 5 个框架验证准确性"

建议：v0.0.19 工具层是架构级决策，如果用户对单 agent 报告存疑，可采用方案 D（成本最低，10 分钟内出复核结果）。

二、各框架工具层深度分析

2.1 一线 AI 编程/Agent 工具

Claude Code

工具通过 buildTool() 工厂构造，每个工具有 inputSchema（Zod，双重用途：生成 JSON Schema 给 API + 运行时 safeParse 校验）、call()、isConcurrencySafe(input)（输入相关并发判定，ls -la 安全 / rm -rf 不安全）、checkPermissions()、validateInput()。

fail-closed 默认：新工具默认串行、视为写操作、跳过自动安全分类器。14 步执行管线 checkPermissionsAndCallTool()：4 步校验 → 2 步准备 → 3 步权限（PreToolUse 钩子 → 权限裁决 → 拒绝处理）→ 5 步执行（含 PostToolUse 钩子与错误处理）。

7 级权限模式：default / acceptEdits / plan / dontAsk / bypassPermissions / auto / bubble。

结果预算化：每个工具声明 maxResultSizeChars，超限落盘到 ~/.claude/tool-results/{hash}.txt 并以预览替换，会话级聚合预算防止"千刀万剐"。

工具延迟加载：shouldDefer: true 仅发送 name + description，需 ToolSearch 调用才加载完整 schema。MCP 工具与内置工具通过 assembleToolPool() 合并，内置优先 + MCP 其后，保持 prompt-cache 断点。40+ 工具分 5 类：File（Read/Edit/Write/NotebookEdit）、Search（Glob/Grep）、Execution（Bash/PowerShell/Monitor）、Web（WebSearch/WebFetch）、Code Intelligence（LSP）。

OpenAI Codex CLI

Rust 实现，ToolHandler trait 定义 kind()/matches_kind()/is_mutating()/handle()，ShellHandler 处理 Function 与 LocalShell 两种 payload。ToolOrchestrator 协调"审批 → 沙箱 → 执行"三段。ShellRequest 携带 command/cwd/timeout/env/proxy/sandbox 权限/justification。

沙箱分层：Unix 用 landlock/seccomp，Windows 用 restricted token + 文件系统 ACL carving；SandboxType 由 SandboxManager::transform 根据 fs/network 策略决定。

审批缓存：ApprovalKey 哈希，同会话内相同命令不再追问。优雅提权：沙箱拒绝时可重试为非沙箱并提示用户。unified_exec.rs 用 PTY 持久 shell 会话，支持 ZshFork 模式。输出 capped 至 EXEC_OUTPUT_MAX_BYTES。

OpenAI Agents SDK

五类工具：Hosted OpenAI 工具（WebSearch/FileSearch/CodeInterpreter/ImageGen/HostedMCP/ToolSearch）、本地运行时工具（Computer/ApplyPatch/Shell/LocalShell）、Function calling（@function_tool）、Agents as tools（agent.as_tool）、Experimental（codex_tool）。

@function_tool 装饰器从函数签名 + docstring 通过 griffe + pydantic 自动生成 JSON Schema。关键参数：strict_mode、is_enabled、needs_approval、failure_error_function、timeout/timeout_behavior、defer_loading、tool_input/output_guardrails。RunContextWrapper 作为可选首参注入应用上下文（不发给模型）。可返回 ToolOutputText/Image/FileContent 多模态结构化输出。

Cursor

内置 12 个工具：Shell、Read、Write、StrReplace、Grep（ripgrep）、Glob、SemanticSearch、Browser、Task（子 agent）、WebSearch、WebFetch、MCP。

多模型分工：Main Agent LLM + Embedding LLM + Re-ranking LLM + Apply Model（cheap 代码专用，把语义 diff 写成实际文件）+ Summarization LLM。

沙箱默认：Shell 写权限限 workspace、网络限已知包管理器、部分 syscall 受限；AI 可请求 full_network/all 提权但需用户审批。MCP 配置在 ~/.cursor/mcp.json。无状态循环：每次请求都重建上下文，用 prompt caching 缓解成本。

Aider

无沙箱、无传统 function calling。Coder 工厂按 edit_format 实例化子类（search-replace/diff/whole-file/udiff）。两类文件集：abs_fnames（可编辑）+ abs_read_only_fnames（只读上下文），自动检测文件提及 + glob 展开 + .aiderignore。

Commands 类实现 slash 命令：/add、/drop、/read-only、/run（! 前缀执行 shell）、/test（非零退出码才入 chat）、/git、/commit。通过 litellm 支持任意 LLM 含本地 Ollama。

Devin

Brain（无状态云端推理）+ Devbox（Firecracker microVM 执行）双层架构。Devbox 三件套：shell、browser、code editor，Brain 通过递归 agentic loop（Plan-Act-Observe-Correct）驱动。

Firecracker 硬件级隔离强于 Docker（不共享 host kernel），每个 session 独立 microVM，VM 跨 session 持久。CLI 4 级权限：Normal/Accept Edits/Bypass/Autonomous（配 --sandbox）。网络按域名 allow/deny list 过滤。

GitHub Copilot Agent Mode

内置工具 read_file/edit_file/run_in_terminal 等 + system prompt 指示 LLM 持续迭代到完成态。custom agent 通过 .agent.md + YAML frontmatter 定义。预置 MCP server：github（只读）+ playwright。VS Code 三类工具：built-in / MCP / extension，3 级审批 Default/Bypass/Autopilot，单请求最多 128 工具。Copilot CLI 内置子 agent：explore/task/general-purpose/code-review/research/rubber-duck。

2.2 通用 Agent 框架

LangChain

@tool 装饰器：函数 docstring → description，类型注解 → input schema。@tool("name") 改名，@tool(args_schema=MyBaseModel) 用 pydantic 定义复杂 schema，@tool(return_direct=True) 短路 agent loop。StructuredTool.from_function 程序化创建。InjectedToolArg/InjectedToolCallId 注解隐藏参数，ToolRuntime 直接注入 state/context/store。返回类型：string / dict / multimodal / Command。错误处理：返回错误字符串而非抛异常，handle_tool_error=True。

LangGraph

ToolNode 预置节点处理并行执行 + 错误处理 + 状态注入。ToolRuntime[Context, State] 注入工具访问 state/context/store/stream_writer/config/tool_call_id。普通工具返回 ToolMessage，命令工具返回 Command（更新 state / 触发导航 / 发消息）。三层 context：static runtime context / dynamic state / store（跨会话）。

AutoGPT (Forge)

组件化架构取代旧 plugin 系统。AgentComponent 基类 + protocol（DirectiveProvider/CommandProvider/MessageProvider/AfterParse/ExecutionFailure/AfterExecute）。命令用 @command 装饰 + JSON Schema 参数。内置 11 个组件：System、UserInteraction、FileManager（local/GCS/S3）、CodeExecutor（Docker）、ActionHistory、GitOperations、ImageGenerator、WebSearch、WebSelenium、Context、Watchdog。Forge 已 legacy，active 开发转向 AutoGPT Platform（可视化 graph builder）。

CrewAI

BaseTool 子类（带状态/校验/env_vars/sync+async）或 @tool("Name") 装饰器（简单工具）。args_schema: Type[BaseModel] 用 pydantic 强校验。_run() 同步 / _arun() 异步。cache_function 自定义缓存。env_vars 声明所需环境变量。

Microsoft AutoGen

FunctionTool 类包装 Python 函数，类型注解 + description 自动生成 JSON schema。ToolUseAgent 持 ChatCompletionClient + Tool 列表，asyncio.gather 并行执行所有 tool calls，结果包成 FunctionExecutionResultMessage 回喂模型反思。ToolAgent 专用执行 agent，抛 ToolNotFoundException/InvalidToolArgumentsException/ToolExecutionException。内置工具：PythonCodeExecutionTool（Docker）、LocalSearchTool/GlobalSearchTool（GraphRAG）、mcp_server_tools、HttpTool、LangChainToolAdapter。

Smolagents

核心创新：CodeAgent——LLM 写 Python 代码片段调用工具，而非 JSON tool call。支持函数嵌套/循环/条件，benchmark 显示比 JSON 方式少 30% 步骤且性能更高。ToolCallingAgent 是传统 JSON blob 方式。Tool 基类 + @tool 装饰器 + 类属性 name/description/inputs/output_type。工具来源：Tool.from_hub/from_langchain/from_space、ToolCollection.from_mcp（Stdio/Streamable HTTP/SSE）。执行器：LocalPythonExecutor（非沙箱）/ Docker / E2B / Modal / Blaxel。模型无关。

Pydantic AI

注册方式：@agent.tool（需 RunContext）/ @agent.tool_plain（纯函数）/ tools=[...] / toolsets=[...]。griffe 解析 docstring。Toolset 抽象是核心：AbstractToolset 含 get_tools/call_tool/get_instructions/for_run/for_run_step/__aenter__/__aexit__。13+ 内置 toolset 类型：FunctionToolset/CombinedToolset/FilteredToolset/PrefixedToolset/RenamedToolset/PreparedToolset/ApprovalRequiredToolset/DeferredLoadingToolset/IncludeReturnSchemasToolset/SetMetadataToolset/WrapperToolset/ExternalToolset/MCPToolset，链式方法 .filtered().prefixed().approval_required().defer_loading()。高级：tool search 延迟加载、prepare 动态过滤、ModelRetry 显式重试、并行 tool call（sequential=True 屏障）、per-tool 超时、人在回路审批。

LlamaIndex

FunctionTool 继承 AsyncBaseTool，from_defaults() 自动从函数 + docstring（Sphinx/Google/Javadoc）+ 类型注解生成 fn_schema（pydantic），过滤 Context/self 参数。同步/异步双向自动转换。ToolOutput 含 blocks（ContentBlock 列表：Text/Image/Audio）。FunctionAgent 用 llm.astream_chat_with_tools()，scratchpad 存 workflow Context，支持 return_direct 短路。

Semantic Kernel

@kernel_function 标注函数，SDK 自动扫描生成 KernelParameterMetadata + JSON schema。auto function calling 6 步循环：序列化 → 发模型 → 模型返回 chat 或 function call → 提取 name/params → invoke → 结果回喂 → 重复直到 chat 或 max iterations。支持并行 function call。保留参数名 kernel/service/execution_settings/arguments 自动注入不广告给 LLM。三种 plugin 导入：Native code / OpenAPI spec / MCP Server。建议：≤20 工具（最好 ≤10）、snake_case 命名、最少参数、return type 简单。

MetaGPT

Action 基类 + Role 基类 Observe-Think-Act loop。三种 react_mode：REACT/BY_ORDER/PLAN_AND_ACT。标准 SOP 5 角色 ProductManager → Architect → ProjectManager → Engineer → QaEngineer，对应 Action：WritePRD/WriteDesign/WriteTasks/WriteCode/WriteCodeReview/WriteTest/RunCode/DebugError。ActionNode 树状 prompt schema + 动态 pydantic 模型生成 + fill 策略 + review/revision。Function calling 通过 aask_code + GENERAL_FUNCTION_SCHEMA（execute(language, code)），4 种响应解析回退（标准 tool call → JSON decode err 回退 regex → markdown code block → plain text）。

2.3 桌面/本地 LLM 工具

Jan

桌面端 Tauri（Rust）+ TypeScript，本地 llama.cpp，OpenAI 兼容 server @ 1337。Jan Server（Go）微服务架构：Kong Gateway + LLM API/Response API/MCP Tools(8091)/Memory Tools(8090)/Realtime API。MCP Tools Service 代理 Serper/SearXNG/scraping/file search/SandboxFusion。Response API 迭代 tools/list + tools/call，受 RESPONSE_MAX_TOOL_DEPTH 与 TOOL_EXECUTION_TIMEOUT 约束。

LM Studio

OpenAI 兼容 /v1/chat/completions tools 数组。原生 tool use（hammer 徽章）：Qwen2.5/Llama-3.1/3.2/Mistral。其他模型走 fallback [TOOL_REQUEST]...[END_TOOL_REQUEST] 自定义 system prompt。MCP 0.3.17+ 引入，0.4.0+ API 支持：ephemeral MCP（per-request）+ pre-configured ~/.lmstudio/mcp.json。工具调用确认弹窗 + allowed_tools 限制。Python/TypeScript SDK .act() 自动多轮 tool calling，异常自动回喂 LLM。推荐 Qwen2.5-7B-Instruct。

Ollama

v0.3.0 引入 tool calling。Go template 驱动：用 text/template 格式化 tool 定义进 prompt，parser 用状态机从流式输出搜模型特定 tag（[TOOL_CALLS] Qwen / 不同格式 DeepSeek/Mistral）。Python SDK 可直接传函数，自动从 docstring + type hints 解析为 schema。流程：messages + tools → 模型返回 tool_calls → client 执行 → 结果作 tool role 消息追加 → 再调模型。支持单 shot / 并行 / 多轮 agent loop / 流式。OpenAI 兼容 + Anthropic 兼容（含 Web Search Loop）。推荐 llama3.1:8b（8/10）/ qwen2.5:14b（9/10）。最佳实践：description < 200 字、6-8 工具上限、tool_choice 强制、max_turns 防失控、try/except 返回结构化错误 JSON。

GPT4All

C++20 + Qt6/QML + llama.cpp + ggml，GPU via Vulkan/Metal/CUDA。LLModel 抽象基类 + ChatLLM（管理对话状态 + 在独立 Qt 线程解析 tool call）+ LocalDocs（SQLite 嵌入式向量 + BM25 混合搜索，全离线 RAG）+ Server（QHttpServer OpenAI 兼容端点）。原生 C++ Jinja2 兼容 chat template 引擎。Python binding + OpenAI 兼容本地 server（Docker）+ Code Interpreter（设备端 JS 沙箱）。Tool calling 在 ChatLLM 层已有 parser，但 chat UI 全功能 tool 支持仍在社区讨论（#3599），倾向 MCP。

AnythingLLM

@modelcontextprotocol/sdk + StdioServerTransport，三层架构：MCP Protocol Layer → Tool Layer → Client Layer。39+ 工具双路路由：11 core tools（workspace/chat/document/system settings）+ 28 additional tools。CallToolRequestSchema 两阶段：解析 → 校验 → 路由 switch → 执行 → 包装 → 错误处理。错误统一 {content:[{type:'text',text:...}],isError:true}，catch-all 保证 server 不崩。PR #3547（2025-03）为原生 AnythingLLM 带来完整 MCP：默认 mcp_config.json + 动态重载 + Agent Skills 管理 MCP + 内置 agent 框架调用运行中 MCP + npx/uv/uvx Docker 支持。5 个 agent 管理 tool，最小权限（tools 数组白名单）+ 持久 registry + workspace 隔离。

OpenWebUI

两类扩展层：In-Process Python（Tools/Pipe/Filter/Action/Skills）+ External HTTP（Native MCP HTTP / MCPO 代理 stdio→HTTP / OpenAPI Servers）。Tools 脚本持有 server 端密钥（user 和 model 都读不到）。Native Mode（v0.6.31+，唯一支持模式）用结构化 tool call 不破坏 KV cache，自动注入 30+ 系统工具。MCP 原生仅支持 Streamable HTTP（多租户匹配 / 进程管理 / 并发冲突 / 容器限制 / 安全），stdio 通过 mcpo 桥接（uvx mcpo --port 8000 -- your_mcp_server，自动生成 FastAPI + Swagger）。OAuth 2.1 + PKCE + DCR + 每用户独立认证。9400+ 公开 MCP server（2026-04，YoY 7.8x）。

MCP（协议本身）

client-host-server 架构 + JSON-RPC 2.0。Host 管多 client 实例 + 安全策略 + 用户授权；Client 1:1 有状态 session 与 Server；Server 暴露 resources/tools/prompts。设计原则：server 极易构建 / 高可组合 / 不能读整个对话或窥其他 server / 通过 capability negotiation 渐进加功能。两层：Data Layer（JSON-RPC lifecycle + tools/resources/prompts + sampling/elicitation/logging）+ Transport Layer（stdio 本地 / Streamable HTTP 远程）。Tool 定义：name(1-128 字符 [A-Za-z0-9_-\.]) / title / description / inputSchema(JSON Schema 2020-12) / outputSchema / annotations。结果：非结构化（content 数组）或结构化（structuredContent 符合 outputSchema）。两类错误：Protocol Error（JSON-RPC error）/ Tool Execution Error（isError:true，LLM 可据以行动）。多轮请求：InputRequiredResult + inputRequests + requestState。

三、模式归纳（跨框架共性）

3.1 工具注册：四大主流范式

  1. 装饰器 + 函数签名协议：LangChain @tool、CrewAI @tool、Smolagents @tool、AutoGen FunctionTool、Pydantic AI @agent.tool、LlamaIndex FunctionTool.from_defaults、Q-agent @tool——签名即协议，inspect.signature + pydantic 自动 schema 是事实标准。
  2. 基类继承 + pydantic args_schema：CrewAI BaseTool、Smolagents Tool、AutoGPT AgentComponent——适合有状态/复杂校验工具。
  3. 工厂函数 + Zod/JSON Schema：Claude Code buildTool()、OpenAI Agents SDK FunctionTool(...) 直构、MCP server inputSchema——schema 即单一真相源。
  4. 方法标注 + 自动发现：Semantic Kernel @kernel_function、MetaGPT @register_tool、AutoGPT @command——SDK 扫描注册。

跨框架共识：docstring → description + 类型注解 → schema 自动生成是标配；pydantic（Python 生态）或 Zod（TS 生态）是 schema 首选；description 质量 > 命名。

3.2 安全：四档光谱

  • 无沙箱派：Aider、MetaGPT、LangChain/LangGraph、Q-agent（永不沙箱 + 黑名单 + 路径保护）——靠输入校验 + 文件分级 + 用户审批。
  • 轻约束派：Claude Code（7 级权限 + fail-closed + 钩子）、Cursor（workspace 写限 + 包管理器网络白名单）、LM Studio（确认弹窗 + allowed_tools）。
  • OS 沙箱派：OpenAI Codex CLI（landlock/seccomp + Windows restricted token + ACL carving）、Devin（Firecracker microVM）——硬件级/内核级隔离。
  • 容器沙箱派：AutoGPT CodeExecutor（Docker）、AutoGen PythonCodeExecutionTool（Docker）、Smolagents（E2B/Docker/Modal/Blaxel）——代码执行走容器。

3.3 工具调用接口：三格式鼎立

  1. OpenAI function calling（tools 数组 + tool_calls 响应 + tool role 回喂）：事实标准。
  2. Anthropic tool_use block：Claude Code、Ollama Anthropic 兼容层。
  3. Prompt 解析（自定义标记）：LM Studio fallback [TOOL_REQUEST]、OpenWebUI legacy mode（已废弃）、Q-agent dispatcher 解析 ```json ... ``` 块、MetaGPT markdown code block 回退、Smolagents CodeAgent（写 Python 代码）。

KV cache 友好性是当代分水岭：OpenWebUI 专为此废弃 legacy prompt injection 模式转向 Native Mode。Q-agent dispatcher 的 json block 解析属 prompt 注入路径，需评估 cache 影响。

3.4 结果回喂：四策略

  • 截断阈值：Codex EXEC_OUTPUT_MAX_BYTES、Claude Code maxResultSizeChars、Q-agent ContextManager L1 落盘。
  • 落盘 + 占位符：Claude Code 落盘 ~/.claude/tool-results/{hash}.txt + 预览替换（最精细）。
  • 会话级预算：Claude Code 聚合 conversation budget 防"千刀万剐"——独有。
  • 直接回喂：大多数框架无特殊处理。

3.5 错误处理：三范式

  • 结构化错误回喂 LLM：Ollama 最佳实践、MCP isError:true、AutoGen ToolExecutionException——主流。
  • 重试机制：Pydantic AI ModelRetry + per-tool retry limit、AutoGen 反思循环、Smolagents self-heal。
  • PermissionError 单独处理：Claude Code/Codex/Devin——权限错误需区别于普通执行错误。

3.6 持久化 / 并发 / MCP / 本地 LLM

  • sqlite：Q-agent / GPT4All LocalDocs / OpenWebUI
  • asyncio.gather 并行：AutoGen / Pydantic AI / OpenAI Agents SDK / LangGraph ToolNode
  • QThread：Q-agent ChatWorker / GPT4All ChatLLM
  • MCP 原生首等公民：Claude Code / Cursor / Codex / OpenAI Agents SDK / LM Studio / Jan / OpenWebUI / AnythingLLM / VS Code Copilot
  • MCP Adapter 挂载：LangChain / LangGraph / LlamaIndex / Smolagents / Pydantic AI / Semantic Kernel / AutoGen / CrewAI
  • 本地 LLM 原生：Ollama / LM Studio / GPT4All / Jan / OpenWebUI
  • 本地 tool calling 双轨制：原生 function calling（Qwen2.5/Llama 3.1/Mistral）+ prompt 解析 fallback——标配

四、对 Q-agent 的启示

4.1 Q-agent 当前设计定位

  • @tool(name, desc) 装饰器 + 函数签名协议 + inspect.signature 注册校验——优势完全契合主流装饰器 + 签名协议范式
  • safety 黑名单 + 路径保护（G:\agent 禁区）+ 永不沙箱——优势属"无沙箱派"，与 Aider/MetaGPT/LangChain 同阵营
  • dispatcher 从 LLM 输出解析 ```json ... ``` 块——风险属 prompt 解析路径，对支持原生 tool calling 的本地模型是退步
  • ContextManager 4 级压缩 + L1 工具结果落盘——优势比多数框架精细，接近 Claude Code
  • sqlite3 持久化——优势比 LangChain/AutoGen 默认不入库更扎实

4.2 v0.0.19 工具层实化可借鉴的设计点

  1. Ollama 原生 tool calling 双轨制（最高优先级）：原生 tool_calls 路径（KV cache 友好）+ prompt 解析 fallback
  2. 结构化错误回喂：PermissionError / 执行错误 / 超时 三类分别返回结构化 JSON 让 LLM 区分行动
  3. 工具结果预算 + 落盘：会话级聚合预算防"千刀剐"
  4. 并发安全标识：@tool 装饰器可选参数
  5. 保留 call_id 关联：tool_result 回喂时 id 关联正确
  6. 危险命令黑名单 + 路径保护已到位：补 PermissionError 回喂格式即可

六、用户应该提但没提的问题（决策清单）

⚠ 这些问题 v0.0.19 落地前必须决策

调研报告里没有覆盖、但工具层实化前必须回答的问题。每个问题的决策都会影响代码结构、UI 行为或安全边界。

6.1 工具命名与描述规范

  1. 工具命名规范：snake_case / camelCase / 中文命令名？跨框架主流是 snake_case（Semantic Kernel 建议），但中文命令名对本地 LLM 友好度更高？
  2. 工具描述质量标准：Ollama 最佳实践是 description < 200 字、6-8 工具上限——Q-agent 是否照搬？描述要不要包含"何时用 / 何时不用 / 参数约束 / 返回格式"四要素？
  3. 工具描述语言：中文描述 vs 英文描述？本地 LLM（Qwen 中文优化）对中文描述理解更好，但 MCP 国际化场景英文更通用
  4. 工具分类前缀：是否参考 Claude Code 5 类前缀（file_ / search_ / exec_ / web_ / code_）？还是平铺命名？

6.2 调用超时与执行控制

  1. 工具调用超时策略：shell 长命令（如 build、apt install）超时多少秒？是否允许用户配置？默认 30s / 60s / 120s？
  2. 工具调用深度上限：防 doom loop——单 turn 最多调几次工具？Jan 用 RESPONSE_MAX_TOOL_DEPTH，Ollama 用 max_turns，Q-agent 编排层已有 doom_loop 检测，阈值定多少？
  3. 工具失败的 LLM 自愈重试上限：失败几次后放弃？Pydantic AI per-tool retry limit——Q-agent 定几次？
  4. 工具执行并发 vs 串行：Q-agent 当前 dispatcher 顺序执行，v0.0.19 是否引入并行？参考 AutoGen asyncio.gather / Pydantic AI sequential=True 屏障

6.3 结果治理与持久化

  1. 工具结果落盘位置：~/.q-agent/tool-results/ 还是项目目录内？跨会话是否保留？清理策略（7 天 / 30 天 / 手动）？
  2. 工具结果会话级预算：Claude Code 用聚合 conversation budget 防"千刀剐"，Q-agent 单工具截断 + 总预算双重控制——总预算定多少 token？
  3. 工具调用日志持久化粒度：完整 input/output 都记？还是只记元数据（tool_name + 谗传时间 + 成功/失败）？记入 sqlite 还是 jsonl？
  4. 工具调用历史是否可回放：用户重启后能不能看到上次的工具调用记录？

6.4 用户审批与权限分级

  1. 用户审批机制：是否引入"危险工具调用前弹窗确认"（参考 Cursor/LM Studio/Pydantic AI ApprovalRequiredToolset）？还是全部自动执行？
  2. 工具权限分级：read_only / write / destructive 三级？哪级需要用户确认？write_file 算 write 级，move 算 destructive 级？
  3. 审批缓存：同会话内相同命令不再追问（参考 Codex ApprovalKey 哈希）？还是每次都问？
  4. 永不沙箱边界确认：用户是否需要明确签署"永不沙箱风险知情同意书"才能用 shell/python_eval？

6.5 UI 反馈与可观测性

  1. 工具调用的 UI 反馈：用户在 chat 气泡里能不能看到工具调用过程（Claude Code 风格折叠展开）？还是只看最终结果？
  2. 工具调用实时状态：执行中（旋转图标）/ 成功（绿勾）/ 失败（红叉）/ 取消（灰圈）四种状态如何展示？
  3. 工具调用输入输出展示：完整展示？截断展示？点击展开？落盘后展示"结果已落盘到 xxx，预览：..."？
  4. 工具调用气泡的颜色 / 图标：是否参考 Claude Code 的工具气泡样式（灰底 + 工具图标 + 折叠）？

6.6 撤销与安全

  1. 撤销机制：用户能否撤销 write_file / move 等不可逆工具的执行结果？是否走 git 兜底（Aider 模式）？
  2. 私有数据保护：防止 LLM 通过工具泄露敏感数据（如把 .env 内容写入公共文件）——是否引入"敏感文件读取拦截"扩展 safety？
  3. 工具调用的审计日志：合规需求？记不记完整 input/output 用于事后追查？是否涉及用户隐私？
  4. 网络工具的 URL 白名单：http_get/web_fetch 是否限 HTTPS？是否需要域名白名单（参考 Devin）？

6.7 架构与扩展

  1. 工具描述如何注入 system prompt：每轮全量注入 vs 按需加载？token 预算怎么控？
  2. 工具版本管理：工具升级后，旧会话回放兼容性？是否需要工具 schema 版本号？
  3. 工具调用的事务性：多个工具调用原子性？比如"read A + write B + move C"中途失败如何回滚？
  4. 工具注册器的外部 tool 源接口：v0.0.19 的 @tool 注册器是否预留"可注入外部 tool 源"接口（为 v0.0.21+ MCP 适配铺路）？

6.8 调研方法本身

  1. 单 agent 报告是否够用：当前报告是单 agent 一次产出，是否要派多 agent 并行交叉审查提高可信度？
  2. 是否需要源码级复核：单 agent 只查了官方文档 + GitHub README，没读源码细节，关键设计点（如 dispatcher 解析逻辑）是否要派 agent 读源码验证？
  3. 调研结论的时效性：AI 框架迭代快（如 MCP 协议 2025-06-18 → 2025-11-25 已两次更新），本次报告时效性多久？半年后是否需要重做？

调研总结

调研方式：单 agent 一次性产出（非多 agent 协作），覆盖 24 个框架 + 80+ 条权威源。

纳入 v0.0.19 的工具：P0 文件操作（read_file/write_file/list_dir/move/search）+ P1 shell（execute_shell+safety）+ P2 网络（http_get/web_fetch，增量）。

放弃/暂缓的工具：P3 代码执行（永不沙箱冲突，最高风险）+ MCP 适配（v0.0.21+ 中期）+ 工具延迟加载（暂不急需）+ 并发判定/审批缓存（中高端配置）+ PreToolUse/PostToolUse 钩子 + 多模态工具输出。

关键未决问题：32 个问题分 8 大类（命名/超时/结果治理/审批/UI/撤销/架构/调研方法），其中 6.1/6.2/6.4/6.5 必须在 v0.0.19 落地前决策。

核心建议：v0.0.19 实化的最关键动作是补 Ollama 原生 tool calling 双轨制 + 结构化错误回喂 LLM + 工具结果预算，P0 文件 + P1 shell 优先落地，P3 代码执行因永不沙箱约束建议暂缓，MCP 留作 v0.0.21+ 中期适配。
