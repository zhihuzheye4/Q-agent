一、调研背景与方法

Q-agent 当前 v0.0.17 已落地对话/交互层与 LLM 抽象层（本地 Ollama 部分），但编排/规划层（orchestrator/planner，即编排器/规划器）、技能体系、工具调用层、运行期记忆仍是 stub（占位空壳）。软件实质仍是 Ollama 聊天前端，未实现"让 LLM 指令被软件理解并执行"这一核心定位。

本次调研目的：在设计 Q-agent 编排层前，参考当前主流 AI 工具的实际实现，避免闭门造车。调研对象为四个开源或可获取源码的 AI 工具：

项目仓库语言定位编排层核心位置
Claude CodeNonSmiIe/claude-code-source-code-full（反编译镜像）TypeScript终端 AI 编程助手src/QueryEngine.ts + src/query.ts + src/coordinator/
OpenClawopenclaw/openclawTypeScript 91%个人 AI 助手（多通道）src/agents/embedded-agent-runner/run.ts + src/context-engine/
Codexopenai/codexRust 96%终端轻量 coding agentcodex-rs/core/src/tools/orchestrator.rs + codex-rs/core/src/session/turn.rs
OpenCodesst/opencodeTypeScript 68%开源 AI coding agentpackages/opencode/src/session/prompt.ts + processor.ts

调研方法：

  1. sparse-checkout（稀疏检出）只拉各项目编排层相关目录，避免完整 clone 大仓占用磁盘
  2. 派 4 个子 agent 并行拆解，每个 agent 负责一个项目，按统一 12 项维度产出拆解报告
  3. 本报告基于 4 份拆解报告（共 125KB / 1131 行），结合 Q-agent 约束（Python + 本地 LLM + 永不沙箱 + 单人桌面应用）筛选借鉴清单，去其糟粕，取其精华

四个项目源码已落地至 G:/agent/市场调研/01-ClaudeCode/ 至 04-OpenCode/，拆解报告存于 05-拆解报告/，本报告存于 06-架构设计/。

二、四项目编排层共性发现

四个项目语言不同、定位不同、规模不同，但编排层呈现五个共性。这些共性是行业共识，Q-agent 应该遵循。

共性 1：无独立 Planner 模块

四个项目都没有一个叫 planner 的独立模块。意图解析全部由 LLM 的 tool_calling（工具调用）能力承担——LLM 直接在响应中输出结构化的 tool_use（工具使用块）JSON，Orchestrator 直接消费。

  • Claude Code：QueryEngine.ts 内部 queryLoop() 直接拿 LLM 输出的 tool_use 块当意图，needsFollowUp（需后续轮）是唯一循环续跑信号
  • OpenClaw：靠 LLM tool-calling + 多层 stream wrapper（流包装器）清洗非标准输出，没有 Planner
  • Codex：ToolRouter::build_tool_call（router.rs:113-160）+ plan mode 的 PlanModeStreamState（turn.rs:1374-1490）共担规划职责，不是独立模块
  • OpenCode：编排意图以 SubtaskPart（子任务块）/ CompactionPart（压缩块）形式嵌入消息流，由 LLM tool-call + 用户 @mention + overflow 检测共同产生
对 Q-agent 的直接启示
不要做独立 Planner 模块。让 LLM 直接输出 tool_call，Orchestrator 直接消费。这是最简且最符合行业实践的做法。

共性 2：主循环都是 while-true + LLM 调用 + 工具执行 + 结果回喂 + 终止判定

四个项目的 Orchestrator 主循环结构高度一致：

while true:
    if should_stop(state): break        # 终止判定
    response = llm.call(messages, tools) # LLM 调用
    assistant_msg, tool_calls = parse(response)
    messages.append(assistant_msg)
    if not tool_calls: break             # LLM 自然停止
    for tc in tool_calls:
        result = execute_tool(tc)        # 工具执行
        messages.append(tool_result_msg(tc.id, result))  # 结果回喂

差异只在终止条件的精细度、工具执行的并发度、错误回喂的格式。Q-agent 直接照搬这个骨架。

共性 3：上下文压缩是多级的

四个项目都不让消息历史无限增长，都有一套压缩机制。复杂度差异极大：

项目压缩级别触发条件
Claude Code6 级（toolResultBudget → snip → microcompact → contextCollapse → autocompact → reactiveCompact）token 预算 + 413 错误
Codex2 级（auto-compaction + manual）90% 上下文窗口
OpenCode1 级（SessionCompaction）+ tail 保留ctx.needsCompaction
OpenClaw14+ 种 retry 触发条件含多种 compactiontimeout / overflow / silent error 等

Claude Code 的 6 级压缩是 Anthropic 服务端复杂场景的产物，对 Q-agent 单人桌面应用是过度设计。Q-agent 用 2 级即可：简单截断 + 摘要压缩。

共性 4：错误不抛出循环，转 tool_result 块回喂

四个项目都不在工具失败时抛异常中断循环，而是把错误转成 tool_use_error（工具使用错误）块作为 tool_result（工具结果）回喂 LLM，让 LLM 自己决定下一步（重试 / 换工具 / 放弃）。

  • Claude Code：tool_use_error 块回喂 + yield synthetic assistant message（产出合成助手消息）应对 API 错误
  • Codex：sandbox（沙箱）denial 转结果回喂 + GuardianRejectionCircuitBreaker（审查者拒绝熔断器）
  • OpenCode：Effect.retry（重试）+ SessionRetry.policy 处理 5xx / rate limit（速率限制）
  • OpenClaw：14+ 种 retry 触发条件分类处理
对 Q-agent 的直接启示
工具失败不抛异常，生成 ToolResult(error=msg) 回喂 LLM。LLM API 失败时产出合成 assistant 消息告知出错 + 终止当前 turn（单轮）。

共性 5：终止条件是多 bool / 多 enum 组合

没有一个项目用单一条件终止循环。终止靠多个布尔/枚举组合表达：

  • Claude Code：11 种终止原因集中在 Terminal（终止态）类型
  • Codex：needs_follow_up + token_limit_reached + has_pending_input 三 bool 组合
  • OpenCode：MAX_STEPS_PROMPT（步数软截止）+ doom_loop（死循环检测，连续 3 次相同工具相同输入）+ ctx.needsCompaction
  • OpenClaw：分级 retry + idle-timeout breaker（空闲超时熔断）+ tool loop detection（工具循环检测）

Q-agent 应采用多 bool 组合：user_cancel（用户取消）+ llm_stopped（LLM 自然停止）+ max_steps_reached（达步数上限）+ consecutive_errors（连续错误超阈值）+ context_overflow（上下文溢出）。

三、Q-agent 约束过滤

共性已明，但不能照抄。Q-agent 有自身约束，必须按约束过滤借鉴清单。

Q-agent 特殊性

  1. 永不沙箱（CLAUDE.md 第五节）：Codex 的 sandboxing（沙箱）/ guardian（审查者）/ execpolicy（执行策略）整个守护链不适用
  2. 本地 LLM 优先：OpenCode 的 20+ provider 抽象层 + OpenClaw 的多 backend（CLI/ACP/embedded）不适用，Q-agent 只需 Ollama + 云端 stub
  3. 单人桌面应用：OpenClaw 的多通道（WhatsApp/Telegram/Slack 等）+ gateway（网关）+ daemon（守护进程）+ cron（定时任务）+ multi-agent routing（多代理路由）全部不适用
  4. Python 实现：Codex 的 Rust 强类型 trait + OpenCode 的 Effect 框架 + Claude Code 的 TypeScript 都不适用，需要 Python 等价物（dataclass + typing.Protocol）
  5. 贴纸式开发原则（CLAUDE.md 第二十一节）：编排层要做成独立模块，不侵入既有 UI/LLM/工具层
  6. 零第三方依赖起步：不引入 Effect / SQLite / pydantic 等运行时依赖，只用标准库

四、设计原则
  1. LLM 直接输出意图：不做独立 Planner 模块。LLM 通过 Ollama tool_calling 直接输出结构化工具调用，Orchestrator 直接消费
  2. 错误不抛出循环：工具失败转 ToolResult(error=...) 回喂 LLM；LLM 失败产出合成 assistant 消息告知 + 终止当前 turn
  3. 状态整体覆写：每次循环迭代结束 state = TurnState(**next_fields) 整体替换，防跨迭代状态泄漏（借鉴 Claude Code）
  4. 多 bool 终止组合：循环终止靠多个布尔值组合判断，不靠单一条件
  5. 软截止优先于硬中断：达 max_steps 时先给 LLM 一条 prefill 提示"请总结收尾"，让 LLM 自然收尾；连续错误超阈值才硬中断
  6. 本地 LLM 防循环：doom_loop 检测连续 3 次相同工具相同输入强制终止（借鉴 OpenCode）
  7. 贴纸式挂载：编排层独立模块，不侵入既有 UI/LLM/工具层，通过信号槽或回调连接
  8. 接口预留多 agent：当前只做单 agent，但 ToolCall 数据类型预留 delegate_agent（委派代理）种，未来扩展不破坏架构
  9. 零第三方依赖：编排层只用标准库（dataclass + typing.Protocol + threading），不引入 pydantic / Effect / asyncio 框架
  10. 可测试性优先：每个模块可独立单测，Orchestrator 用 mock LLM + mock tool 测试，不依赖真实 Ollama
五、模块划分

编排层位于 q_agent/orchestrator/ 下，分为 5 个模块。所有模块独立，对外通过 __init__.py 暴露最小接口。

q_agent/orchestrator/ ├── __init__.py # 暴露 Orchestrator + 公开类型 ├── types.py # 数据类型（Message/ToolCall/ToolResult/TurnState/TerminationReason） ├── loop.py # Orchestrator 主循环（run_turn 函数） ├── dispatcher.py # LLM 输出解析 + 工具调用调度 ├── context.py # 上下文管理（历史存储 + 压缩） └── turn.py # 单轮 turn 抽象 + 状态机

六、核心数据类型

所有数据类型放 types.py，用 @dataclass（数据类）定义，零第三方依赖。

# q_agent/orchestrator/types.py
from __future__ import annotations
from dataclasses import dataclass, field
from enum import Enum
from typing import Literal, Optional


class Role(str, Enum):
    """消息角色"""
    SYSTEM = "system"      # 系统提示
    USER = "user"          # 用户消息
    ASSISTANT = "assistant"  # 助手消息
    TOOL = "tool"          # 工具结果消息


@dataclass
class ToolCall:
    """工具调用意图（来自 LLM 输出的 tool_use 块）"""
    id: str                        # 工具调用 ID（LLM 生成，用于配对 tool_result）
    name: str                      # 工具名（如 read_file / shell_exec）
    arguments: dict[str, object]   # 工具参数（LLM 输出的 JSON 解析结果）


@dataclass
class ToolResult:
    """工具执行结果（回喂给 LLM 作为 tool_result 块）"""
    call_id: str                   # 对应 ToolCall.id
    content: str = ""               # 成功内容
    error: Optional[str] = None     # 失败时的错误信息（二选一）


@dataclass
class Message:
    """统一消息结构（送入 LLM 和存入历史）"""
    role: Role
    content: str = ""                              # 文本内容（user/assistant/system）
    tool_calls: list[ToolCall] = field(default_factory=list)  # assistant 触发的工具调用
    tool_call_id: Optional[str] = None            # tool 角色消息对应的 ToolCall.id
    is_synthetic: bool = False                    # 合成消息标记（错误时产出，非 LLM 真实输出）


class TerminationReason(str, Enum):
    """循环终止原因"""
    LLM_STOPPED = "llm_stopped"               # LLM 自然停止（无 tool_call 输出）
    USER_CANCEL = "user_cancel"               # 用户取消
    MAX_STEPS_REACHED = "max_steps_reached"    # 达到最大步数（软截止后仍不收尾）
    CONSECUTIVE_ERRORS = "consecutive_errors"  # 连续错误超阈值
    CONTEXT_OVERFLOW = "context_overflow"      # 上下文溢出无法压缩
    DOOM_LOOP = "doom_loop"                    # 死循环检测（连续 N 次相同工具相同输入）
    LLM_FAILED = "llm_failed"                 # LLM API 多次重试失败


@dataclass
class TurnState:
    """单轮状态（每次循环迭代结束整体覆写，防跨迭代泄漏）"""
    messages: list[Message]                   # 当前消息历史（含本轮新增）
    steps: int = 0                             # 已执行步数
    consecutive_errors: int = 0                # 连续错误计数
    last_tool_signature: Optional[tuple[str, str]] = None  # (tool_name, args_json) 用于 doom_loop 检测
    doom_loop_count: int = 0                  # 连续相同工具调用计数
    terminated: bool = False                   # 是否已终止
    termination_reason: Optional[TerminationReason] = None  # 终止原因


@dataclass
class TurnResult:
    """单轮结果（run_turn 返回）"""
    messages: list[Message]                   # 最终消息历史
    final_assistant_text: str                 # 最终 assistant 文本回复
    termination_reason: TerminationReason      # 终止原因
    steps_executed: int                        # 实际执行步数
    tool_calls_made: list[ToolCall]            # 本轮所有工具调用
设计要点
  • TurnState（单轮状态）每次迭代用 state = TurnState(**next_fields) 整体替换，防字段跨迭代泄漏（借鉴 Claude Code）
  • is_synthetic（合成标记）标识非 LLM 真实输出的占位消息，UI 渲染时可用不同样式（如红色错误气泡）
  • TerminationReason（终止原因）枚举给 UI 明确反馈，用户知道是 LLM 自然停止还是被中断
  • last_tool_signature（上次工具签名）用于 doom_loop 检测，连续 3 次相同工具相同输入强制终止
七、主循环设计（Orchestrator）

主循环位于 loop.py，是编排层核心。借鉴四项目共识的 while-true 骨架 + Q-agent 自身约束。

Orchestrator 类定义

# q_agent/orchestrator/loop.py
from __future__ import annotations
from typing import Callable, Optional
from q_agent.llm import LLMClient
from q_agent.orchestrator.types import (
    Message, TurnState, TurnResult, TerminationReason, ToolCall, ToolResult
)
from q_agent.orchestrator.context import ContextManager
from q_agent.orchestrator.dispatcher import parse_response, execute_tool_calls


class Orchestrator:
    """编排器主循环驱动"""

    def __init__(
        self,
        llm_client: LLMClient,
        context_manager: ContextManager,
        max_steps: int = 30,
        consecutive_error_threshold: int = 3,
        doom_loop_threshold: int = 3,
        on_step: Optional[Callable[[TurnState], None]] = None,
    ) -> None:
        self.llm = llm_client
        self.ctx = context_manager
        self.max_steps = max_steps
        self.err_threshold = consecutive_error_threshold
        self.doom_threshold = doom_loop_threshold
        self.on_step = on_step  # UI 回调，每步触发（用于进度展示）

    def run_turn(self, user_input: str) -> TurnResult:
        """主循环：从用户输入到最终 assistant 回复"""
        # 1. 初始化：用户消息进上下文
        self.ctx.append_user_message(user_input)
        state = TurnState(messages=list(self.ctx.messages))

        while True:
            # 2. 终止判定（多 bool 组合）
            reason = self._check_termination(state)
            if reason is not None:
                return self._finalize(state, reason)

            # 3. 调用 LLM（流式）
            try:
                stream = self.llm.chat_stream(
                    messages=self.ctx.to_llm_format(),
                    tools=self.ctx.available_tools(),
                )
                assistant_msg, tool_calls = parse_response(stream)
            except Exception as e:
                # 4. LLM 失败：产出合成 assistant 消息 + 计数 + 继续判断
                synth = Message(
                    role=Role.ASSISTANT,
                    content=f"[LLM 调用失败：{e}]",
                    is_synthetic=True,
                )
                self.ctx.append_message(synth)
                state = TurnState(
                    messages=list(self.ctx.messages),
                    steps=state.steps,
                    consecutive_errors=state.consecutive_errors + 1,
                    last_tool_signature=state.last_tool_signature,
                    doom_loop_count=state.doom_loop_count,
                )
                continue

            # 5. assistant 消息入历史
            self.ctx.append_message(assistant_msg)

            # 6. LLM 自然停止（无 tool_call）→ 终止
            if not tool_calls:
                state = self._advance(state, assistant_msg, tool_calls, 0)
                return self._finalize(state, TerminationReason.LLM_STOPPED)

            # 7. 工具执行（含错误回喂）
            results = execute_tool_calls(tool_calls)
            for result in results:
                tool_msg = Message(
                    role=Role.TOOL,
                    content=result.content if result.error is None else result.error,
                    tool_call_id=result.call_id,
                )
                self.ctx.append_message(tool_msg)

            # 8. doom_loop 检测：连续相同工具相同输入
            new_sig = self._tool_signature(tool_calls)
            is_doom = new_sig == state.last_tool_signature
            doom_count = state.doom_loop_count + 1 if is_doom else 0

            # 9. 步数推进 + 状态整体覆写
            state = TurnState(
                messages=list(self.ctx.messages),
                steps=state.steps + 1,
                consecutive_errors=0,  # 成功执行清零
                last_tool_signature=new_sig,
                doom_loop_count=doom_count,
            )

            # 10. UI 回调（流式进度）
            if self.on_step:
                self.on_step(state)

    def _check_termination(self, state: TurnState) -> Optional[TerminationReason]:
        """终止判定（多 bool 组合）"""
        if state.consecutive_errors >= self.err_threshold:
            return TerminationReason.CONSECUTIVE_ERRORS
        if state.doom_loop_count >= self.doom_threshold:
            return TerminationReason.DOOM_LOOP
        if state.steps >= self.max_steps:
            # 软截止：先 prefill 提示 LLM 收尾，下一轮再判断
            # 简化版：直接终止（v0.0.18 不做 prefill，留 v0.0.19+）
            return TerminationReason.MAX_STEPS_REACHED
        if self.ctx.is_overflow():
            return TerminationReason.CONTEXT_OVERFLOW
        return None

    def _tool_signature(self, tool_calls: list[ToolCall]) -> tuple[str, str]:
        """工具调用签名（用于 doom_loop 检测）"""
        if not tool_calls:
            return ("", "")
        first = tool_calls[0]
        import json
        return (first.name, json.dumps(first.arguments, sort_keys=True))

    def _advance(self, state, msg, tool_calls, err_delta):
        """状态推进（整体覆写）"""
        return TurnState(
            messages=list(self.ctx.messages),
            steps=state.steps,
            consecutive_errors=state.consecutive_errors + err_delta,
            last_tool_signature=state.last_tool_signature,
            doom_loop_count=state.doom_loop_count,
        )

    def _finalize(self, state, reason) -> TurnResult:
        """收尾：提取最终 assistant 文本"""
        final_text = ""
        for m in reversed(state.messages):
            if m.role == Role.ASSISTANT and not m.is_synthetic:
                final_text = m.content
                break
        return TurnResult(
            messages=state.messages,
            final_assistant_text=final_text,
            termination_reason=reason,
            steps_executed=state.steps,
            tool_calls_made=[tc for m in state.messages for tc in m.tool_calls],
        )

循环步骤详解

步骤动作失败处理
1. 初始化用户消息入 ContextManager（上下文管理器）—
2. 终止判定多 bool 检查（错误数/doom/步数/溢出）命中即返回 TurnResult
3. LLM 调用流式调 chat_stream，parse_response 解析异常 → 产出合成 assistant 消息 + 错误计数+1 + 继续判断
4. assistant 入历史ContextManager.append_message—
5. LLM 自然停止tool_calls 为空 → 直接 finalize—
6. 工具执行dispatcher.execute_tool_calls 批量执行失败转 ToolResult(error) 回喂
7. tool_result 入历史每条工具结果作为 role=TOOL 消息—
8. doom_loop 检测新签名 vs 上次签名 → 计数或清零达阈值即终止
9. 状态整体覆写TurnState(**new_fields) 替换—
10. UI 回调on_step(state) 触发—
八、Planner 决策：不做独立 Planner

这是本架构与"教科书 Agent 架构"最大的差异点。本节给出理性论证。

教科书架构 vs 实际项目

教科书 Agent 架构常画成：User → Planner → Orchestrator → Tool → LLM（用户 → 规划器 → 编排器 → 工具 → LLM），Planner 负责把用户意图拆成多步计划，Orchestrator 负责执行计划。

但四个主流项目的实际实现都没有独立 Planner。原因：

  1. 现代 LLM 已内化规划能力：通过 tool_calling，LLM 自己决定调用什么工具、传什么参数。不需要外置 Planner 拆任务
  2. 外置 Planner 反而约束 LLM 灵活性：LLM 可以根据工具执行结果动态调整下一步，外置 Planner 的固定计划反而僵化
  3. 节省 token：Planner 模块通常要再调一次 LLM 做规划，等于两倍 LLM 调用。直接让执行 LLM 输出 tool_call 是单次调用
  4. 代码简洁性：少一个模块，少一份接口，少一份测试

Q-agent 的 Planner 在哪里？

"规划"职责被分摊到三个地方：

  1. 系统提示（在 ContextManager）：告诉 LLM 可用工具、调用格式、约束
  2. LLM 自身：通过 tool_calling 输出结构化意图
  3. dispatcher.parse_response（在 dispatcher.py）：解析 LLM 输出的 tool_use 块成 ToolCall 对象

这三个加起来等于 Planner 职责，但不需要独立模块。

什么时候才需要独立 Planner？

以下场景未来若出现，再考虑引入独立 Planner：

  • 需要多步前瞻规划（如 chess 那种搜索树展开）
  • 用户输入是模糊意图，需要 LLM 先澄清再执行
  • 需要根据用户历史行为预测意图（个性化规划）
  • 需要把任务拆给多个子 agent 并行执行（多 agent 委派）

当前 Q-agent 单 agent + 单 Ollama + 单用户场景都不属于以上任何一种，所以不做独立 Planner。

架构预留
如果未来需要 Planner，可在 dispatcher.py 加 pre_plan(messages) -> list[ToolCall] 钩子，在调 LLM 前先注入规划。无需改动 Orchestrator 主循环。
九、工具调用循环

工具调用循环在 dispatcher.py，负责把 LLM 输出的 tool_use 块解析成 ToolCall 列表，调度 q_agent.tools.executor 执行，收集结果。

dispatcher.py 接口

# q_agent/orchestrator/dispatcher.py
from __future__ import annotations
from typing import Iterator
from q_agent.orchestrator.types import Message, ToolCall, ToolResult, Role
from q_agent.tools.executor import execute_tool, execute_shell
from q_agent.tools.safety import check_command, check_path


def parse_response(stream: Iterator) -> tuple[Message, list[ToolCall]]:
    """解析 LLM 流式响应 → assistant 消息 + 工具调用列表

    Args:
        stream: LLMClient.chat_stream() 返回的流式响应

    Returns:
        (assistant_msg, tool_calls)
        - assistant_msg: 包含 LLM 文本输出 + tool_use 块
        - tool_calls: 解析出的 ToolCall 列表（可能为空）
    """
    # Ollama /api/chat 流式输出 NDJSON，每行含 message.delta
    # 累积文本 + 累积 tool_calls（Ollama tool_calls 格式：message.tool_calls）
    text_parts: list[str] = []
    tool_calls: list[ToolCall] = []

    for chunk in stream:
        if chunk.get("message", {}).get("content"):
            text_parts.append(chunk["message"]["content"])
        if chunk.get("message", {}).get("tool_calls"):
            for tc in chunk["message"]["tool_calls"]:
                tool_calls.append(ToolCall(
                    id=tc["id"],
                    name=tc["function"]["name"],
                    arguments=tc["function"]["arguments"],
                ))
        if chunk.get("done"):
            break

    assistant_msg = Message(
        role=Role.ASSISTANT,
        content="".join(text_parts),
        tool_calls=tool_calls,
    )
    return assistant_msg, tool_calls


def execute_tool_calls(tool_calls: list[ToolCall]) -> list[ToolResult]:
    """执行工具调用列表，收集结果（含错误回喂）

    顺序执行（v0.0.18 不做并发，避免本地资源竞争）。
    失败不抛异常，转 ToolResult(error) 回喂 LLM。
    """
    results: list[ToolResult] = []
    for tc in tool_calls:
        try:
            # 安全校验（不违反永不沙箱原则，仅基本输入校验）
            _validate_safety(tc)

            # 调用 q_agent.tools.executor 执行
            if tc.name == "shell_exec":
                output = execute_shell(tc.arguments.get("cmd", ""))
            else:
                output = execute_tool(tc.name, tc.arguments)

            results.append(ToolResult(call_id=tc.id, content=str(output)))
        except Exception as e:
            # 工具失败 → 转 ToolResult(error) 回喂，不抛异常
            results.append(ToolResult(call_id=tc.id, error=f"{type(e).__name__}: {e}"))
    return results


def _validate_safety(tc: ToolCall) -> None:
    """基本安全校验（不是沙箱）"""
    if tc.name == "shell_exec":
        cmd = tc.arguments.get("cmd", "")
        check_command(cmd)   # 危险命令黑名单拦截
    if tc.name in ("read_file", "write_file", "move_file"):
        path = tc.arguments.get("path", "")
        check_path(path)     # 项目根目录保护

工具调用循环时序

用户输入 ↓ [LLM 调用] ← ContextManager.to_llm_format() ↓ parse_response(stream) ├─→ assistant_msg (含 text + tool_use 块) └─→ tool_calls: [ToolCall(id, name, args), ...] ↓ ContextManager.append_message(assistant_msg) ↓ [LLM 自然停止? tool_calls 为空] 是 → finalize(LLM_STOPPED) 否 ↓ execute_tool_calls(tool_calls) ├─→ ToolResult(content=...) 成功 └─→ ToolResult(error=...) 失败（不抛异常） ↓ 每个 ToolResult → Message(role=TOOL) → ContextManager.append_message ↓ [doom_loop 检测] [步数推进] [状态整体覆写] ↓ 回到 [LLM 调用] 进入下一轮
十、上下文管理（分级压缩）

位于 context.py。Q-agent 采用 2 级压缩，比 Claude Code 6 级简单，比 OpenCode 1 级更稳健。

ContextManager 类

# q_agent/orchestrator/context.py
from __future__ import annotations
from q_agent.orchestrator.types import Message, Role


class ContextManager:
    """消息历史 + token 计数 + 分级压缩"""

    def __init__(
        self,
        system_prompt: str,
        max_tokens: int = 8000,        # 本地 LLM 上下文窗口上限
        soft_limit_ratio: float = 0.8,  # 软阈值（达 80% 触发压缩）
        keep_recent: int = 6,           # 压缩时保留最近 6 条消息
    ) -> None:
        self.system_prompt = system_prompt
        self.max_tokens = max_tokens
        self.soft_limit = int(max_tokens * soft_limit_ratio)
        self.keep_recent = keep_recent
        self.messages: list[Message] = []

    def append_user_message(self, text: str) -> None:
        self.messages.append(Message(role=Role.USER, content=text))

    def append_message(self, msg: Message) -> None:
        self.messages.append(msg)
        # 达软阈值触发压缩（二级：摘要压缩）
        if self.estimate_tokens() > self.soft_limit:
            self._compact()

    def to_llm_format(self) -> list[dict]:
        """转 Ollama /api/chat 格式"""
        result = [{"role": "system", "content": self.system_prompt}]
        for m in self.messages:
            result.append(self._msg_to_dict(m))
        return result

    def available_tools(self) -> list[dict]:
        """返回工具元数据（供 LLM tool_calling）"""
        # v0.0.18 占位：从 q_agent.tools 注册表拉取
        from q_agent.tools import list_tools
        return list_tools()

    def estimate_tokens(self) -> int:
        """粗略 token 估算（1 token ≈ 4 字符，本地 LLM 中文更省）"""
        total_chars = sum(len(m.content) for m in self.messages)
        return total_chars // 4

    def is_overflow(self) -> bool:
        """硬溢出（无法压缩回软阈值以下）"""
        return self.estimate_tokens() > self.max_tokens

    def _compact(self) -> None:
        """二级压缩：摘要旧消息 + 保留最近 N 条 + 系统提示不动

        策略（借鉴 OpenCode tail 保留 + Claude Code microcompact）：
        1. 保留最近 self.keep_recent 条消息原样
        2. 较老消息合并成一条 summary Message
        3. 系统提示不动
        """
        if len(self.messages) <= self.keep_recent * 2:
            return  # 消息不够多，跳过

        # 分块：旧消息 + 最近消息
        old_msgs = self.messages[:-self.keep_recent]
        recent_msgs = self.messages[-self.keep_recent:]

        # 生成摘要（v0.0.18 简化版：用文本截断，不调 LLM 做摘要）
        # 真摘要留 v0.0.19+：调 LLM 自己生成 summary
        summary_text = self._summarize(old_msgs)

        summary_msg = Message(
            role=Role.ASSISTANT,
            content=f"[上下文摘要] {summary_text}",
            is_synthetic=True,
        )

        self.messages = [summary_msg] + recent_msgs

    def _summarize(self, msgs: list[Message]) -> str:
        """v0.0.18 简化版：取每条消息前 100 字符拼成摘要
        v0.0.19+ 改为调 LLM 生成真摘要"""
        parts = []
        for m in msgs:
            if m.role == Role.USER:
                parts.append(f"用户问: {m.content[:100]}")
            elif m.role == Role.ASSISTANT and not m.is_synthetic:
                parts.append(f"AI答: {m.content[:100]}")
            elif m.role == Role.TOOL:
                parts.append(f"工具结果: {m.content[:100]}")
        return " | ".join(parts)[:500]  # 总长不超 500 字

压缩级别说明

级别触发动作v0.0.18 实现
L1 软阈值压缩token 达 80% max摘要旧消息 + 保留最近 6 条简化版文本截断
L2 硬溢出终止压缩后仍超 max循环终止 + 用户提示is_overflow() 返回 True
未来 L3 真摘要同 L1调 LLM 生成 summaryv0.0.19+ 预留
v0.0.18 简化策略
v0.0.18 用文本截断做摘要（每条消息取前 100 字符），不调 LLM 生成真摘要。理由：① 本地 LLM 调用慢，压缩时再调一次会拖慢 ② 单人桌面应用对话一般不超过 50 轮，截断足够 ③ 真摘要留 v0.0.19+ 优化。
十一、错误处理与降级

借鉴 Claude Code"错误永不抛出循环"+ OpenCode retry policy，Q-agent 错误处理分四类：

错误分类与处理

错误类型来源处理方式循环影响
工具执行失败q_agent.tools.executor 抛异常转 ToolResult(error) 回喂 LLM不中断，LLM 自决策下一步
LLM API 失败Ollama 连接断/超时/500产出合成 assistant 消息 + consecutive_errors+1不中断，下次循环重试
安全校验失败safety.check_command/check_path 拒绝转 ToolResult(error="命令被安全策略拒绝")不中断，LLM 知道被拒
上下文溢出压缩后仍超 max_tokens循环终止 + 用户提示"上下文过长"终止当前 turn

降级策略

# 错误回喂示例

# 工具失败
try:
    output = execute_tool(tc.name, tc.arguments)
    result = ToolResult(call_id=tc.id, content=str(output))
except Exception as e:
    result = ToolResult(
        call_id=tc.id,
        error=f"{type(e).__name__}: {e}",
    )
    # 不抛异常，继续回喂

# LLM 失败（在 loop.py 主循环）
try:
    stream = self.llm.chat_stream(...)
    assistant_msg, tool_calls = parse_response(stream)
except Exception as e:
    synth = Message(
        role=Role.ASSISTANT,
        content=f"[LLM 调用失败：{e}]",
        is_synthetic=True,
    )
    self.ctx.append_message(synth)
    state = TurnState(
        messages=list(self.ctx.messages),
        steps=state.steps,
        consecutive_errors=state.consecutive_errors + 1,
        last_tool_signature=state.last_tool_signature,
        doom_loop_count=state.doom_loop_count,
    )
    continue  # 下次循环重试

连续错误阈值

consecutive_errors >= 3 时循环终止（TerminationReason.CONSECUTIVE_ERRORS）。理由：本地 LLM 不稳定时连续失败 3 次基本无救，强试浪费时间，不如让用户重启或换模型。

十二、循环终止条件

终止靠多 bool 组合，对应 TerminationReason 枚举。每个终止原因 UI 都要能反馈给用户。

原因触发条件UI 反馈优先级
LLM_STOPPEDLLM 输出无 tool_call，自然停止正常显示 assistant 文本1（最高）
USER_CANCEL用户点击取消按钮气泡 [已取消]1
DOOM_LOOP连续 3 次相同工具相同输入气泡 [检测到循环，已终止]2
CONSECUTIVE_ERRORS连续 3 次 LLM 失败气泡 [LLM 连续失败，请检查模型]2
MAX_STEPS_REACHED步数达 30气泡 [达到步数上限]3
CONTEXT_OVERFLOW压缩后仍超 max_tokens气泡 [上下文过长，请新建对话]3
LLM_FAILEDLLM 首次调用就失败错误气泡2

软截止 vs 硬中断

借鉴 OpenCode MAX_STEPS_PROMPT 软截止模式：

  • v0.0.18 简化版：达 max_steps 直接终止（硬中断）
  • v0.0.19+ 改进：达 max_steps 时给 LLM 一条 prefill 提示"请在下一条消息中总结收尾"，让 LLM 自然收尾；如果再不收尾才硬中断
用户取消实现
借鉴 v0.0.17 取消按钮接通 ChatWorker.stop 的模式（ADR-031）：Orchestrator.run_turn 启动时记录 stop_flag，每轮迭代开头检查 stop_flag，命中即返回 USER_CANCEL。UI 同步立即清理不等后台线程。

class ContextManager:
    def __init__(self, memory_store: MemoryStore):
        self.store = memory_store  # v0.0.20+ 接入

    def append_message(self, msg: Message) -> None:

贴纸式约束
编排层只能引用其他层的公开接口，不允许被其他层反向引用。任何对 LLMClient / executor / ChatPage 类继承、objectName、公开接口的改动 = 架构级改动，必须先出方案让用户否决（CLAUDE.md 第二十一节）。
十五、实施路线（三步走）

编排层落地分三个版本，每版本都是独立可测单元，最后一步接通 UI。

v0.0.18：编排层骨架 + 完整单测

目标：编排层 5 模块全部落地 + 单测覆盖 80%+，但 UI 看不到变化（用 mock LLM + mock tool 测试）。

  1. 实现 types.py 数据类型
  2. 实现 turn.py TurnState 状态机
  3. 实现 dispatcher.py parse_response + execute_tool_calls
  4. 实现 context.py ContextManager（含简化压缩）
  5. 实现 loop.py Orchestrator 主循环
  6. 测试：mock LLM 返回 tool_use 块 + mock executor 返回结果，覆盖所有路径（正常/工具失败/LLM 失败/doom_loop/max_steps/取消）
  7. 打 v0.0.18 .exe（编排层代码进包，UI 仍走 ChatWorker 不接 Orchestrator）

预计工作量：3-5 个工作日。

v0.0.19：工具调用层实化 + 编排层接通

目标：把 stub 的工具调用层实化（@tool 装饰器 + executor + safety 真工作），编排层 dispatcher 真调 executor，端到端跑通。

  1. 实化 q_agent/tools/：@tool 装饰器 + 内置工具（echo / read_file / write_file / shell_exec）+ executor + safety
  2. 编排层 dispatcher.execute_tool_calls 改为真调 executor（v0.0.18 是 mock）
  3. 编排层 dispatcher.parse_response 改为真解析 Ollama 输出（v0.0.18 是 mock chunk）
  4. 集成测试：mock Ollama HTTP server + 真工具执行，跑端到端流程
  5. 打 v0.0.19 .exe（编排层 + 工具层代码进包，UI 仍走 ChatWorker）

预计工作量：3-5 个工作日。

v0.0.20：UI 接通 + 用户可见能力

目标：UI 层 ChatWorker 改为调 Orchestrator，用户在 UI 上能看到"LLM 调用了 read_file 工具，结果如下"。

  1. 新建 q_agent/ui/orchestrator_worker.py：QThread 包 Orchestrator.run_turn，发 step/done/failed 信号（贴纸式挂载，不侵入 ChatWorker）
  2. ChatPage 加 _on_orchestrator_step slot：收到 step 信号时在气泡里追加工具调用提示
  3. ChatPage 改 chat 触发：从直接调 ChatWorker 改为先调 OrchestratorWorker，再 fallback 到 ChatWorker（兼容模式）
  4. 工具调用 UI：气泡里显示 [调用工具: read_file(path=G:\test.txt)] + [结果: ...]
  5. 打 v0.0.20 .exe（第一个用户可见编排层能力的版本）

预计工作量：5-7 个工作日。

风险与回滚

关键风险
  • v0.0.18 单测用 mock 跑通，v0.0.19 接真 LLM 真工具时可能出现 mock 未覆盖的边界情况 → 回滚到 v0.0.18 mock 模式，定位问题
  • v0.0.20 UI 接通可能影响 v0.0.17 的取消按钮等既有功能 → 必须按贴纸式原则独立挂载 OrchestratorWorker，不侵入 ChatWorker，可随时回滚
  • 本地 LLM 输出格式可能与 Ollama 标准 NDJSON 有差异 → dispatcher.parse_response 需要容错
