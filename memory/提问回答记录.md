#### 1. PySide6 + PyInstaller 是什么

**PySide6**：Qt 6 框架的官方 Python 绑定（由 Qt 公司官方维护，LGPL 协议，商用免费）。它做两件事：

**PyInstaller**：把 Python 程序打包成独立 .exe 的工具。它做的事是：把 Python 解释器 + 所有第三方库 + 代码 + 资源文件全部塞进单个 .exe。用户拿到 .exe 双击即可运行，**不需要装 Python 或任何依赖**。这就是 CLAUDE.md 第二十节"分发零外部依赖硬规则"的技术基础——靠 PyInstaller `--onefile` 实现一个文件交付完整软件。

#### 2. 既然没装 PySide6，为什么脚本能生成矢量图

关键区分：**SVG 本质是 XML 文本文件**，不是位图。Python 写 SVG 文件就是写一段 XML 字符串到磁盘，**完全不需要任何图形库**——纯文本 IO。

脚本默认模式 `python scripts/generate_icons.py` 只生成 SVG（XML 文本），所以不需要 PySide6。

PySide6 只在两种场景下才需要：
- 跑 `python scripts/generate_icons.py --png`（用 QtSvg 把 SVG 渲染成 PNG 位图）
- 实施 UI 界面时（PySide6 主窗口 + 加载 SVG 显示）

本次只跑默认 SVG 模式，所以没装 PySide6 也能跑通，5 个测试 SVG 全部生成成功。

### AI 回答：

三个问题分别回答：

#### 1. Python 不自带 Qt6，需要重新下载——对

Python 标准库只有 `tkinter`（基础 GUI，控件原始），**没有 Qt6**。

Qt6 是独立的 C++ 应用程序框架（由 Qt 公司开发），跟 Python 无关。要在 Python 中用 Qt6，需要装 PySide6（Qt6 的官方 Python 绑定）。

所以：是，需要 `pip install PySide6` 装到 F 盘 venv。

#### 2. Python 自带库不能调用或渲染 SVG，必须用 Qt6——部分对

要把"调用 SVG"分清两层：

**A. 解析 SVG（读/改/写 XML 文本）**：Python 标准库 `xml.etree.ElementTree` 可以做。SVG 本质是 XML 文本，标准库能读、改、写。这次脚本就是这么干的——纯文本 IO，不需要 Qt6。

**B. 渲染 SVG 成位图显示**（在屏幕上画出 SVG 描述的图形，或转 PNG）：Python 标准库**做不到**。SVG 渲染涉及解析路径数据 + 计算几何 + 渐变 + 滤镜 + 光栅化，标准库没这个能力。必须用专门的 SVG 渲染库。

Qt6 通过 QtSvg 模块（`QSvgRenderer`）提供 SVG 渲染能力。**本项目选 Qt6 而不是其他 SVG 库（cairosvg 等）的原因**：Qt6 同时提供 GUI 框架（窗口/控件）+ SVG 渲染——一石二鸟，避免引入两个独立库（cairosvg + 别的 GUI 库）。

所以：渲染 SVG 成位图**必须**用 Qt6 或类似库；生成 SVG 不需要。

#### 3. Qt6 的本质——你的理解需要修正

你说"Qt6 不是单纯依赖库，应该有它自己的图形窗口"——**对**。

但你说"SVG 只是替换现有"或"SVG 需按它定的规矩排版"——**不准确**。让我澄清：

**Qt6 是什么**：完整的**应用程序框架**（C++ 写的，PySide6 是 Python 绑定），包含：
- 窗口系统（`QMainWindow` / `QDialog` / `QWidget`）
- 控件库（按钮/菜单/输入框/列表/表格/树等几十种）
- 事件系统（鼠标/键盘/定时器/信号槽）
- 渲染引擎（`QPainter` 绘 2D 图形）
- 资源系统（图标/图片/字体打包访问）
- **QtSvg 模块**（解析渲染 SVG，是 Qt6 内置模块之一）

**SVG 是什么**：W3C 标准的矢量图格式，**独立于 Qt6**。SVG 规范由 W3C 维护，所有浏览器都支持，Adobe Illustrator 也能编辑。SVG 跟 Qt6 是**消费关系**——Qt6 消费 SVG（解析+渲染），不是 Qt6 定义 SVG。

**它们的关系**：
- Qt6 提供**窗口和控件**（这是软件的"骨架"）
- SVG 是**图标素材**（这是"装饰"）——按钮上的小图标、菜单项前的小图标等
- Qt6 通过 QtSvg 把 SVG 渲染成 `QPixmap`，再设到 `QIcon` 上，然后挂到按钮/菜单

**SVG 在 Qt6 里怎么"按规矩"用**：
- SVG 文件本身遵循 W3C 标准（跟 Qt6 无关，浏览器也能开）
- 但要在 Qt6 里**显示**，需要走 Qt6 的加载流程：
  1. `QSvgRenderer` 加载 SVG 文件
  2. 渲染到 `QPixmap`（首次渲染一次，缓存）
  3. 包成 `QIcon`
  4. 设到 `QPushButton.setIcon()` 或 `QAction.setIcon()` 等

所以**不是 SVG 按 Qt6 规矩排版**，而是 SVG 文件本身按 W3C 标准，Qt6 按自己的方式消费它。本项目 ADR-014 的"方案 C：SVG sprite + QtSvg 首次缓存"就是这个流程。

**类比**：Qt6 像一个画框（窗口）+ 画廊（控件库），SVG 像画作（矢量图）。画作本身遵循美术规范（W3C SVG 标准），画框负责把它挂上去展示。我们写的脚本是"画师"（生成 SVG 文件），Qt6 是"画框+画廊"（展示 SVG）。

### 用户：你是用位图思维的吗 ？可是之前的讨论你不是说位图思维并不是很好，你有更好的思路了吗？为什么又选择了位图？那么到底有没有更好的思路？

### AI 回答：

你说得对，我之前推荐方案 C 时**没看穿 QPixmap 的本质**。让我承认错误并给出真正的更好思路。

#### 先承认错误

方案 C 的关键步骤："UI 启动时 `QSvgRenderer` 加载 sprite 按 id 渲染每个图标到 `QPixmap` 缓存"——**QPixmap 本质就是位图**（在内存中的像素数据）。把 SVG 渲染成 QPixmap 缓存，本质就是"SVG → 位图（在内存里）→ 显示"，只是位图放在内存而不是磁盘 PNG 文件。

所以方案 C 是**位图思维的变体**——比 SVG→PNG 文件好（少一次磁盘 IO + 不存冗余文件），但本质上还是位图。我之前说"矢量源不丢 / 任意缩放无失真"是夸大了——`QPixmap` 缓存的是固定分辨率的位图，缩放时还是要重新渲染或失真。

你的质疑是对的。

#### 真正更好的思路

**方案 D：QIcon 直接接受 SVG + Qt 内部按 size 智能缓存**

核心思路：**让 Qt 按需渲染，不要预先把所有 SVG 渲染成位图缓存**。

具体做法：
```python
icon = QIcon("q_agent/assets/icons/send-active.svg")  # 直接传 SVG 路径
button = QPushButton()
button.setIcon(icon)
button.setIconSize(QSize(16, 16))  # Qt 按此 size 渲染
```

Qt 内部机制：
- QIcon 收到 SVG 路径后，**不立即渲染成位图**
- 当 button 真正要显示图标时（paintEvent 触发），Qt 按当前 size（如 16x16）调用 `QSvgRenderer` 矢量渲染
- Qt **按 size 自动缓存**：同一 size 第二次显示时不重新渲染，用缓存
- 如果 size 变了（高 DPI 显示器、用户调整图标大小），Qt 自动按新 size 重新渲染

**优点**：
- **完全矢量**：矢量源一直用，按当前 size 渲染，任意缩放/DPI 自适应无失真
- **性能 OK**：Qt 内部按 size 缓存，同 size 不重复渲染
- **DPI 自适应**：高 DPI 屏自动按 2x/3x 渲染，清晰锐利
- **零额外依赖**：QtSvg 是 PySide6 内置
- **API 简单**：一行 `QIcon(svgPath)` 替代方案 C 的 sprite + 预缓存一堆 QPixmap

**缺点**：
- 首次某 size 显示时矢量计算略慢（毫秒级，人感无）
- 复杂图标（L5 ai-brain 多色+滤镜）首次渲染略慢于预缓存的 QPixmap，但后续 size 缓存后无感

#### 跟方案 C 对比

| 维度 | 方案 C（QPixmap 预缓存） | 方案 D（QIcon 直接受 SVG） |
|------|----------------------|------------------------|
| 本质 | 内存位图 | 矢量 + Qt 自动 size 缓存 |
| 缩放/DPI | 失真（要重新渲染） | 自适应（Qt 按新 size 渲染） |
| 内存 | 高（所有图标预渲染占内存） | 低（按需渲染，size 缓存小） |
| 首次显示 | 快（已预渲染） | 略慢（矢量计算，毫秒级） |
| 后续显示 | 快（用缓存） | 快（同 size 用缓存） |
| API 复杂度 | 高（sprite + 预缓存 + id 管理） | 低（一行 QIcon） |
| 矢量源保留 | 保留但用不上（用位图缓存） | 真正用上 |

**结论**：方案 D 全面优于方案 C。方案 C 只在"图标极多 + 显示频率极高 + 用户从不调整 size/DPI 不变"的极端场景略胜——本软件不是这种场景。

### Orchestrator 是什么

Orchestrator（编排器）是 Q-agent 五层结构中编排/规划层的核心模块，位于 q_agent/orchestrator/loop.py。它是一个 while-true 主循环驱动器，负责把"用户输入"加工成"最终 assistant 回复"，中间可能穿插多次工具调用与结果回喂。

定位：当前 v0.0.17 对话是 UI 层 ChatWorker 直接调 OllamaClient.chat_stream，没有经过编排层，软件实质仍是 Ollama 聊天前端。Orchestrator 落地后将接管 LLM → 工具 → 结果回喂 → LLM 智能体循环。

关键设计决策：
- 无独立 Planner（四项目共识，规划由 LLM tool_calling + dispatcher 承担）
- 错误不抛出循环（工具失败转 ToolResult(error) 回喂；LLM 失败产出合成 assistant 消息 + 计数+1）
- 状态整体覆写（每次迭代 state = TurnState(**new_fields) 替换，防跨迭代泄漏）
- 多 bool 终止组合（不靠单一条件）

### 如何工作的（主循环 10 步）

1. 初始化：用户消息入 ContextManager
2. 终止判定：多 bool 检查（错误数/doom/步数/溢出），命中即返回 TurnResult
3. LLM 调用：流式调 chat_stream，parse_response 解析。失败 → 产出合成 assistant 消息 + 错误计数+1 + 继续
4. assistant 入历史：ContextManager.append_message
5. LLM 自然停止：tool_calls 为空 → 直接 finalize（LLM_STOPPED）
6. 工具执行：dispatcher.execute_tool_calls 批量执行。失败转 ToolResult(error) 回喂
7. tool_result 入历史：每条工具结果作为 role=TOOL 消息
8. doom_loop 检测：新签名 vs 上次签名 → 计数或清零
9. 状态整体覆写：TurnState(**new_fields) 替换
10. UI 回调：on_step(state) 触发，UI 流式更新气泡

### 一、"级别"是什么意思

"级"指压缩策略的代价分级——按信息损失大小 + 计算成本由低到高排序的压缩手段序列。设计哲学是"先试便宜的，不行再升级到贵的"，避免每次压缩都直接做最贵的 LLM 总结。

重要澄清：HTML 第二节表中 OpenClaw 写的"14+"严格说不是"压缩级别"，而是"retry 触发条件"——14+ 里只有部分是 compaction（timeout/overflow/threshold/manual），其他是 retry 策略（profile rotation/model fallback/empty response 等）。

### 二、Claude Code 6 级压缩每一级

| 级 | 名称 | 做什么 | 代价 | 触发 |
|---|------|------|------|------|
| 1 | toolResultBudget | 按字符预算裁剪超大 tool_result，替换为"content cleared"占位 | 极低 | 每次循环开头 |
| 2 | snip | 移除僵尸消息与失效标记 | 低 | feature flag 门控 |
| 3 | microcompact | 只压缩 COMPACTABLE_TOOLS（Read/Bash/Grep/Glob/WebSearch/WebFetch/Edit/Write）的旧 tool_result，用 [Old tool result content cleared] 占位 | 低 | 每次循环开头 |
| 4 | contextCollapse | 按 git 提交日志折叠段，保留 granular context | 中 | feature flag 门控 |
| 5 | autocompact | 以上都无法压到阈值才触发，调 compactConversation 让 LLM 总结整个对话历史 | 高（一次 LLM 调用） | token 达 effectiveContextWindow - 13000 |
| 6 | reactiveCompact | API 直接返回 prompt-too-long（413）时触发，drain 已暂存的 collapses 后再尝试 LLM 总结 | 高 | 413 错误时 |

核心思想：1-4 级是主动压缩（不调 LLM），5 级是阈值触发，6 级是反应式。熔断保护：MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3。

### 三、OpenClaw 14+ 种 retry 触发条件

14 种：timeout compaction / overflow compaction / profile rotation / model fallback / empty response / reasoning only / incomplete turn / missing assistant / silent error / before-agent-finalize revision / codex-app-server recovery / mid-turn precheck continuation / compaction continuation / post-compaction guard。

附加保护机制：idle-timeout breaker / tool loop detection（5 种模式：generic_repeat/unknown_tool_repeat/known_poll_no_progress/ping_pong/global_circuit_breaker，30/10/20/30 阈值）/ MAX_RUN_LOOP_ITERATIONS / lane task timeout / abort signal。

核心思想：每种 retry 触发条件独立计数，互不污染 retry budget。

与 Claude Code 6 级的本质区别：Claude Code 6 级是纯压缩流水线（同一目标：减 token），按代价由低到高分级；OpenClaw 14+ 是 retry 触发条件矩阵（不同目标：timeout/overflow/rate_limit/empty/incomplete/revision 等不同错误类别），每类独立处理策略。

### 四、当输入一段长文本，最终输出结果是什么

以 Q-agent 2 级压缩为例（用户输入 10000 字 ≈ 2500 tokens，本地 LLM context window 8000 tokens），结构流程：

1. 用户输入 → ctx.append_user_message
2. while True 主循环：
   - _check_termination 多 bool 检查
   - LLM 调用 chat_stream → parse_response
   - ctx.append_message(assistant_msg) 内部触发压缩检查
   - L1 软阈值压缩：tokens > 80% max_tokens → 分块（old_msgs + recent_msgs[-6:]）→ 旧消息生成摘要（v0.0.18 简化版：每条取前 100 字符截断，总长 ≤500 字）→ 重组 [summary_msg] + recent_msgs
   - LLM 自然停止？tool_calls 为空 → finalize(LLM_STOPPED)
   - tool_calls 非空 → execute_tool_calls → tool_result 入历史 → doom_loop 检测 → 状态整体覆写 → UI 回调 on_step
   - continue 下一轮
3. 某轮压缩后仍 > max_tokens → L2 硬溢出终止（is_overflow 返回 True）→ _check_termination 返回 CONTEXT_OVERFLOW
4. _finalize 提取最终 assistant 文本（最后一个非合成 assistant 消息）
5. 返回 TurnResult

最终输出 TurnResult 结构：
- messages: list[Message]（含合成摘要 + 最近 6 条原样 + 最终 assistant 回复）
- final_assistant_text: str（最后一个非合成 assistant 的 content）
- termination_reason: TerminationReason（LLM_STOPPED / CONTEXT_OVERFLOW / MAX_STEPS_REACHED 等）
- steps_executed: int（实际 LLM 调用次数）
- tool_calls_made: list[ToolCall]（本轮所有工具调用）

## 第 N+3 问：为什么不沿用 OpenClaw 的模式

不沿用 OpenClaw 模式的原因来自三个层面冲突：定位冲突 + 约束冲突 + 复杂度成本冲突。

### 一、定位冲突：OpenClaw 是工业级多通道助手，Q-agent 是单人桌面应用

OpenClaw 核心模块（Q-agent 全不需要）：多通道消息系统（WhatsApp/Telegram/Slack/iOS/Tailscale/WebChat 路由）/网关服务器（HTTP/WebSocket+控制台 UI+设备配对+速率限制+CSP）/ACP 代理客户端协议/守护进程 launchd systemd/Cron 定时任务/子代理多 agent 协作（1780 行 spawn+注册表+深度限制+能力继承+孤儿恢复）/多认证配置轮换（多 credential+冷却+OAuth）/CLI 后端（Claude CLI/Gemini CLI 子进程+JSONL 流解析+MCP 回环抓取）/心跳系统/浏览器插件/摄像头插件/图片视频音乐生成/执行轨迹持久化。

OpenClaw 顶层 agentCommand 函数 2471 行，单 run.ts 4202 行，run/attempt.ts 5780 行——这种体量是为工业级多通道场景服务的。Q-agent 砍掉 11 个模块后 OpenClaw 模式 90% 代码量都不该存在。

### 二、约束冲突：OpenClaw 违反 Q-agent 四大硬规则

1. 永不沙箱：OpenClaw 的 sandbox.ts + sandbox-tool-policy.ts 在 Docker 容器跑工具（workspaceAccess: none/ro/rw）+ bash-tools.exec-host-gateway.ts 委托外部 CLI/Docker。Q-agent 明确永不沙箱，仅基本输入校验。
2. 本地大语言模型优先：OpenClaw 假设多提供商+多认证+多降级（20+ provider 抽象+多认证轮换+模型降级+CLI 后端）。Q-agent 本地 Ollama 优先，这些 90% 是死代码。
3. 零第三方依赖起步：OpenClaw 依赖插件 SDK + Effect 框架 + SQLite + 工作线程 + Docker + MCP 集成。Q-agent 编排层只用标准库 dataclass + typing.Protocol + threading。
4. 分发零外部依赖：OpenClaw 的 Docker 沙箱+外部 CLI 子进程+插件加载器要求用户机器预装 Docker/Claude CLI/Node.js。违反"最终用户拿 .exe 双击即跑"硬规则。

### 三、复杂度成本冲突：OpenClaw 模式的代码可读性代价

1. 插件驱动的间接调用成本：大量间接调用+元数据传递，单文件 5000+ 行调度中心成为常态。
2. 14+ 种重试触发的可读性灾难：run.ts 4202 行代码可读性极差。Q-agent 本地场景无提供商速率限制/无认证过期/无多通道投递失败，只需要 5 种终止条件，14+ 在 Q-agent 80% 是死代码。
3. ContextEngine 接口过度抽象：8 个生命周期方法（bootstrap/maintain/ingest/ingestBatch/afterTurn/assemble/compact/dispose），legacy 引擎只是 pass-through 真实逻辑还在 compaction.ts+agent-session.ts。Q-agent v0.0.18 用 ContextManager 一个类直接可测。
4. stream wrapper 多层清洗的浪费：wrapStreamFnSanitizeMalformedToolCalls/wrapStreamFnRepairMalformedToolCallArguments/wrapStreamFnTrimToolCallNames/wrapStreamFnDecodeXaiToolCallArguments 是为非标准大语言模型输出（xAi 编码奇怪字符串/工具名加前缀）设计。Q-agent 默认 Ollama 输出规范，多层清洗是浪费。

### 四、Q-agent 借鉴了 OpenClaw 哪些工程模式（去其糟粕后取其精华）

借鉴点：死循环检测（5 种模式+30/10/20/30 阈值）/多 bool 终止组合（每种 retry 独立计数互不污染预算）/压缩分块+摘要合并+标识符保留指令（IDENTIFIER_PRESERVATION_INSTRUCTIONS 强制保留 UUID/IP/URL）/压缩后守卫（createPostCompactionLoopGuard 防压缩后无限重跑相同工具）/失败分类器（classifyFailoverReason 速率限制/过载/上下文溢出/认证/超时/不完整轮次/中止/未知 8 类）/空闲超时熔断（idle-timeout-breaker 防本地大语言模型卡死烧钱）。

结论：Q-agent 借鉴 OpenClaw 的工程模式层（死循环检测/多 bool 终止/压缩分块/压缩后守卫/失败分类器/空闲超时熔断），但不沿用 OpenClaw 的架构模式层（插件驱动/多后端抽象/多通道/多认证/多代理/守护进程/网关服务器）。

### 五、核心结论

### OpenClaw 上下文压缩模式具体是什么

### 不沿用 OpenClaw 压缩模式的 10 个原因

1. 每次压缩都调 LLM 做摘要，本地 LLM 场景下严重拖慢响应（OpenClaw 假设云端 LLM 快，Q-agent 走 Ollama 本地 LLM 调一次摘要可能十几秒，用户发一句话压缩触发后要等十几秒才继续响应体验不可接受）。Q-agent v0.0.18 用简化版文本截断（每条取前 100 字符，总长 ≤500 字）零 LLM 调用毫秒级完成。v0.0.19+ 真摘要留作未来优化（且会用单独摘要模型避免占用主对话模型）。

2. 4 种压缩触发点在 Q-agent 本地 LLM 场景下大量冗余。超时压缩不需要（Q-agent 超时直接报错+连续错误+1 不触发压缩）/溢出压缩需要但用 is_overflow() 单一阈值检查就够/阈值压缩需要用 80% 软阈值检查/手动压缩 v0.0.18 不做。4 种 → Q-agent 1-2 种就够。

3. ContextEngine 8 生命周期接口在 Q-agent 桌面软件阶段过度抽象。Q-agent 以开发软件的流程和模式开发桌面软件，强调工程产出清晰、代码可维护、单文件可读。OpenClaw ContextEngine 接口是为可插拔上下文引擎+第三方替换+向量检索引擎+SQLite 持久化+LLM 即记忆设计的。OpenClaw 拆解报告自己承认代价是 legacy 引擎只是 pass-through 真实逻辑还在 compaction.ts+agent-session.ts 要换引擎得自己重新实现完整 lifecycle。Q-agent v0.0.18 用 ContextManager 一个类管消息历史+令牌计数+2 级压缩，零接口抽象直接可测。

4. OpenClaw 工作线程模式在 Q-agent 已有 QThread 异步机制下重复。OpenClaw 用 Node.js worker thread 做分块/摘要避免阻塞主循环（单线程 Node.js 设计）。Q-agent 是 PySide6+QThread 桌面应用，v0.0.20 会建 OrchestratorWorker(QThread) 把整个编排循环放到后台线程，不需要在编排循环内再嵌套 worker thread 做压缩。

5. OpenClaw 异步重试 3 次 500-5000ms 抖动退避在本地 LLM 下无意义。OpenClaw 假设云端 LLM 可能临时故障（502/503/速率限制）需要重试+抖动退避避免雪崩。Q-agent 本地 Ollama 要么成功要么失败：进程崩了重试 3 次还是崩应该让用户重启 Ollama/模型加载失败重试 3 次还是失败应该报错/网络问题本地无网络问题。Q-agent 用 consecutive_errors >= 3 直接终止循环，比 OpenClaw 重试 3 次+500-5000ms 抖动更干脆。

6. 标识符保留指令在 Q-agent 单人桌面对话场景下用不到。OpenClaw 标识符保留指令强制保留 UUID/哈希/IP/URL 不被压缩丢失是工业级场景（多通道多会话多代理长期对话涉及复杂任务标识符）的产物。Q-agent 单人桌面对话一般不涉及复杂标识符，简化版文本截断已能保留对话主线。

7. OpenClaw 压缩模式依赖第三方库违反 Q-agent 分发零外部依赖硬规则。OpenClaw 依赖 Node.js 工作线程（Python 无对应）+ Effect 框架（异步重试/流式截断）+ SQLite（会话持久化）。Q-agent 编排层只用 Python 标准库 dataclass+typing.Protocol+threading，最终用 PyInstaller --onefile 打成单个 .exe 用户双击即跑。OpenClaw 模式无法在零第三方依赖约束下复刻，且即便复刻用户拿到 .exe 还要装 Node.js/Docker，违反"安装后通过 Ollama 调用本地 LLM"的清晰分发路径。

8. OpenClaw 压缩模式是为长时多轮多会话工业场景设计 Q-agent 对话规模小。OpenClaw 用户可能在 WhatsApp/Telegram/Slack 多通道多会话长期对话历史可能上千轮必须分块压缩+多块摘要合并+工作线程避免阻塞+标识符保留。Q-agent 单人桌面应用对话一般 10-50 轮关闭即清空（v0.0.18 不持久化），50 轮以内 2 级压缩完全够用。

9. 从达到 agent 强大能力目标看压缩复杂度不是能力核心。Agent 能力来自编排循环（while-true+工具调用+结果回喂）+工具调用层（@tool 装饰器+executor+safety）+技能体系（@skill 装饰器+内置技能库）+运行期记忆（MemoryStore+RuntimeMemory）+多代理委派（delegate_agent 接口预留）。上下文压缩只是保障长对话不崩的辅助机制不是 agent 能力核心。把工程预算投到编排循环+工具调用层+技能体系更有价值。

10. Q-agent 借鉴了 OpenClaw 压缩模式的工程精华。借鉴点：压缩分块+tail 保留最近 N 轮（OpenClaw 按最大令牌分块消息+OpenCode 会话压缩 tail 保留 → Q-agent _compact 旧消息生成摘要+保留最近 6 条）/压缩后守卫（OpenClaw 创建压缩后循环守卫观察工具结果序列检测重复 → Q-agent 死循环检测连续 3 次相同工具相同输入强制终止）/失败分类（OpenClaw 分类失败转移原因 8 类 → Q-agent 终止原因枚举 7 类）。

### 核心结论

Q-agent 不沿用 OpenClaw 上下文压缩模式的根本原因：

1. OpenClaw 模式每次压缩都调 LLM 做摘要本地 LLM 场景下严重拖慢响应
2. OpenClaw 4 种压缩触发点在 Q-agent 本地 LLM 场景下大量冗余
3. OpenClaw ContextEngine 8 生命周期接口在 Q-agent 桌面软件阶段过度抽象
4. OpenClaw 工作线程模式在 Q-agent 已有 QThread 异步机制下重复
5. OpenClaw 异步重试 3 次抖动退避在本地 LLM 下无意义
6. OpenClaw 模式依赖 Effect/SQLite/工作线程违反 Q-agent 分发零外部依赖硬规则
7. OpenClaw 压缩模式是为长时多轮多会话工业场景设计 Q-agent 对话规模小
8. 上下文压缩不是 agent 能力核心工程预算应投到编排循环+工具调用+技能体系

Q-agent 选择借鉴 OpenClaw 压缩工程精华（分块+tail 保留+压缩后守卫+失败分类）+不抄架构外壳（ContextEngine 8 接口+工作线程+异步重试+标识符保留指令），用 Python 标准库+单文件 200 行内 ContextManager 实现，符合桌面软件开发流程清晰简洁可维护的原则。

- 主对话用主力模型（如 Qwen2.5-14B/GLM-4）
- 摘要用单独小模型（如 Qwen2.5-3B）不占用主对话模型内存
- 摘要在独立 SummaryWorker(QThread) 后台跑
- 主对话不阻塞——用户继续对话摘要完成后再合并到上下文
- Ollama 支持同时加载多个模型（OLLAMA_MAX_LOADED_MODELS 调高）

#### 触发阈值

- 70% 软阈值 → 触发第 1-2 级（同步毫秒级）
- 85% 软阈值 → 触发第 3 级（异步几秒到十几秒）
- 100% 硬溢出 → 触发第 4 级（终止当前轮）

#### 标识符保留（采纳 OpenClaw 思路）

第 3 级真摘要时加标识符保留指令：保留文件路径/任务 ID/工具调用 ID/URL/哈希/IP/模型名/版本号。

### 三、修订后论证：OpenClaw 模式在 2000+ 轮场景下哪些仍不沿用、哪些要采纳

#### 仍不沿用（论证仍然成立的部分）

1. ContextEngine 8 生命周期接口在 Q-agent 桌面软件阶段仍过度抽象——用 ContextManager+sqlite3 后端即可
2. 工作线程避免阻塞主循环在 Q-agent 已有 QThread 异步机制下重复——Q-agent 编排循环已在后台线程
3. 异步重试 3 次 500-5000ms 抖动退避在本地大语言模型下仍无意义——摘要失败直接用第 2 级文本截断兜底降级而非重试
4. 超时压缩触发点在本地大语言模型下不需要——超时直接报错+连续错误+1
5. Effect 框架+Node.js worker thread 违反分发零外部依赖——SQLite 可用 Python 标准库 sqlite3 替代 Effect/Node.js worker thread 无 Python 等价物且不需要
6. 隔离降级机制（quarantine）在 Q-agent 单一 ContextManager 下无适用对象

#### 需要采纳（原论证不成立 2000+ 轮下必须采纳）

1. 分块压缩——2000+ 轮历史无法一次塞进大语言模型做摘要必须分块
2. 多块摘要合并——分块后每块单独摘要再多块摘要合并成总摘要
3. 标识符保留指令——2000+ 轮涉及大量文件路径/任务 ID/工具调用 ID/URL/哈希必须强制保留
4. 持久化——2000+ 轮不能关闭即清空。Q-agent 用 Python 标准库 sqlite3 持久化
5. 压缩后守卫——2000+ 轮下压缩后大语言模型更容易重复跑相同工具必须有守卫。Q-agent 已有死循环检测可复用
6. 多级压缩分级——2 级不够必须 4 级。Q-agent 从 2 级升到 4 级
7. 异步摘要避免阻塞——2000+ 轮下摘要耗时长必须异步不阻塞主对话。Q-agent 用 SummaryWorker(QThread)+独立小摘要模型

#### 部分采纳（按需简化）

1. 4 种触发点（超时/溢出/阈值/手动）→ Q-agent 用 3 种（阈值/硬溢出/手动）去掉超时压缩
2. 超时保护包装 → Q-agent 用 QThread 自带的 wait(timeout) 即可
3. 轮后异步重写会话记录 → Q-agent v0.0.18 不做 v0.0.22+ 按需加

### 四、修订后的核心理由

Q-agent 不沿用 OpenClaw 上下文压缩模式的根本原因（2000+ 轮设计要求下修订版）：

1. OpenClaw ContextEngine 8 生命周期接口在 Q-agent 桌面软件阶段仍过度抽象
2. OpenClaw 工作线程模式在 Q-agent 已有 QThread 异步机制下重复
3. OpenClaw 异步重试 3 次抖动退避在本地大语言模型下仍无意义
4. OpenClaw 超时压缩触发点在本地大语言模型下不需要
5. OpenClaw 依赖 Effect 框架+Node.js worker thread 违反分发零外部依赖（SQLite 可用 Python 标准库 sqlite3 替代）
6. OpenClaw 隔离降级机制在 Q-agent 单一 ContextManager 下无适用对象

采纳 OpenClaw 压缩模式工程精华（2000+ 轮下必须采纳）：分块压缩+多块摘要合并+标识符保留指令+持久化（sqlite3 标准库）+压缩后守卫（已有死循环检测可复用）+多级压缩分级（2 级升 4 级）+异步摘要避免阻塞（SummaryWorker QThread+独立小摘要模型）。

Q-agent 选择：借鉴 OpenClaw 压缩工程精华（分块/合并/标识符保留/持久化/压缩后守卫/多级/异步）+不抄架构外壳（ContextEngine 8 接口/工作线程/异步重试/超时压缩/隔离降级/Effect 框架），用 Python 标准库（sqlite3+threading+dataclass）+SummaryWorker(QThread)+独立小摘要模型实现，符合桌面软件安装包分发+Ollama 调用本地大语言模型+达到 agent 强大能力的定位。

**重要设计要求修订记录**：Q-agent 对话规模设计要求从原 50 轮修订为 2000+ 轮。v0.0.18 编排层骨架的压缩策略从原 2 级（软阈值截断+硬溢出终止）修订为 4 级（工具结果预算/旧消息截断/异步真摘要/硬溢出终止）+sqlite3 持久化+SummaryWorker 异步摘要+标识符保留指令。
