# 方案：Ollama Cloud 转发模型分类修复（v0.0.7）

时间：2026-06-20 12:30
状态：已执行（commit 6d631bf）

## 背景

用户准确指出 v0.0.6 之前的分类 bug：minimax-m3 / deepseek-v4-pro 等通过 Ollama Cloud 转发的模型并非真正本地装权重，但旧 list_models() 只取 /api/tags 的 name 字段一律塞"本地"组，没看 remote_model / remote_host 字段。

同时澄清两点：
- CLOUD_PRESET（gpt-4o/claude/gemini）是硬编码占位（ADR-019 明确）
- 本地组是动态调 API（无缓存），用户"删除模型检测不出来"推测不成立——下次启动重新调 /api/tags 会刷新

## 决策

1. 新增 `ModelEntry(NamedTuple)`：name + is_remote + remote_host
2. `list_models()` 破坏性改签名：返回 `list[ModelEntry]`，`is_remote = bool(remote_model) or bool(remote_host)`
3. UI 下拉框改三组结构：本地（真本地）+ Ollama Cloud（转发）+ 云端预置（CLOUD_PRESET 占位）。Ollama Cloud 组仅在存在转发模型时才显示 header
4. graceful degradation：老版 Ollama 缺 remote_* 字段时退化为 v0.0.6 行为，不会更糟
5. CLOUD_PRESET 保留不变（语义不同：通过 Ollama 转发 vs 直连 OpenAI/Anthropic/Google API 未实现）

## 后果

- 优点：分类正确——minimax-m3 等显示在 Ollama Cloud 组而非本地组
- 优点：用户可视觉区分三类模型，下次 Ollama cloud 模型再多也不会再错分
- 优点：graceful——老版 Ollama 不会因缺字段报错
- 取舍：list_models() 签名破坏性变更（list[str] → list[ModelEntry]），调用点仅 toolbar.worker，已同步更新
- 取舍：CLOUD_PRESET 仍是硬编码占位，等真接 API 时改动态拉取（ADR-020+ 范围）
