## 一、用户需求

> v0.0.13 验证后用户严肃批评："增加功能按键就好像在空白的地方贴一张贴纸，能点击这个按钮影响到的代码也只是这个功能相关的代码。当前这个状态和我一开始设计的严重背离。"
>
> 用户阐述设计哲学：UI 所有图片存一个文件夹，软件启动先唤醒 UI 窗口，UI 元素用坐标定位，代码引用方式"当 X 图标被点击 → 执行这些程序"，新增/删除按钮只动该按钮代码不影响其他，按钮之间互不依赖，一个崩溃不引起整个窗口错误。
>
> 用户问这种思路叫什么、有没有更好更成熟的，并明确"我不在乎思路，我只在乎满足我的需求，不需要阿谀奉承"。

## 二、思路定名与选型

- 用户思路叫"绝对坐标布局/画布式 GUI"（VB/Delphi/Win32/Qt Designer 绝对坐标模式）
- 现代等价物"Qt 布局管理器 + 独立 widget 模块"——满足同样隔离需求（按钮独立/增删零连锁/资源集中/回调简单）+ 自动响应式 + 动态内容原生支持
- 资源占用三者几乎相同（差异 < 1KB，被 PySide6 100MB 级开销淹没），选择不该基于资源占用
- 用户确认按"Qt 布局管理器 + 独立 widget + 严格贴纸式"重构

## 三、三层反思（错在哪一层）

### 第一层：直接根因（v0.0.13 验证后图标 + tooltip 仍消失）

- 根因 A：v0.0.12 打包命令漏 `--add-data "q_agent/assets;q_agent/assets"`，SVG 没进 .exe，运行时 importlib.resources 找不到图标，tab 图标全部消失（独立低级错误，跟架构无关但同次出现）
- 根因 B：v0.0.12 sidebar 容器化破坏 QSS 选择器匹配 + 事件层级，导致 tooltip 异常

### 第二层：设计哲学违背

- v0.0.12 把 Sidebar 从 QListWidget 改成 QFrame 容器内嵌 HardwareMonitor，违反"贴纸式"——把新功能塞进既有组件，破坏既有组件内聚性
- 正确做法：sidebar 保持 v0.0.9 QListWidget 子类零改动，HardwareMonitor 独立模块，MainWindow left panel 一行 addWidget 挂载

### 第三层：规则层面缺失

- CLAUDE.md 没明文写"贴纸式开发"原则。第十九节"大决策先展示"是通用规则，但没具体到 UI 模块边界
- AI 违反第十九节——v0.0.12 容器化是架构级改动，应先展示方案让用户否决，但 AI 直接执行了
- 第二个错：打包命令没对照 README.md 模板，手敲漏参数
- 第三个错：测试盲区——只验证开发期 svg 存在，没验证打包后 .exe 内资源完整性

## 四、执行内容

### 4.1 架构回退（治本）

- `q_agent/ui/sidebar.py` 恢复 v0.0.9 QListWidget 子类原貌：
  - `class Sidebar(QListWidget)`，objectName="Sidebar"
  - 4 tab + 图标 + tooltip + tab_changed 信号
  - 零内嵌 HardwareMonitor
  - 保留 SIDEBAR_WIDTH=200

- `q_agent/ui/main_window.py` 新建 left panel 贴纸式挂载：
  - left panel = QWidget + QVBoxLayout（sidebar stretch=1 + hardware_monitor 底部固定 160px）
  - left_panel.setFixedWidth(LEFT_PANEL_WIDTH=200) 放进水平布局左侧
  - `self.sidebar = Sidebar(left_panel)` + `self.hardware_monitor = HardwareMonitor(left_panel)` 两行 addWidget
  - closeEvent 调 `self.hardware_monitor.stop()` 优雅退出
  - QTimer.singleShot(200, self.hardware_monitor.start) 启动 worker

- `q_agent/ui/theme.py` 恢复 QSS 选择器：
  - 4 处 `QListWidget#SidebarList` 改回 `QListWidget#Sidebar`
  - 删除 v0.0.13 新增的 `QFrame#Sidebar` 和 `QListWidget#SidebarList`

- `q_agent/ui/hardware_monitor.py` 保持不变（已是独立模块）

### 4.2 测试改造

- `tests/test_ui_imports.py`：
  - `test_main_window_constructs`：`w.sidebar._list.count()` → `w.sidebar.count()`；`w.sidebar.hardware_monitor` → `w.hardware_monitor`
  - `test_sidebar_tab_switches_stack`：`w.sidebar._list.setCurrentRow(i)` → `w.sidebar.setCurrentRow(i)`

### 4.3 规则固化（防止再犯）

- CLAUDE.md 新增第二十一节"贴纸式开发原则"：
  - 基座（v0.0.1 UI 框架）永不再动（除非明确扩展公开接口）
  - 贴纸 = 新功能独立 widget + 父容器一行 addWidget + 信号槽连接
  - 架构级改动判定：改既有类继承/objectName/公开接口/塞新功能进既有模块内部 → 触发第十九节"先展示方案"
  - 违规自检：每次写代码前自问"这个改动是否动了既有模块的类继承/objectName/公开接口/内部结构？"

- CLAUDE.md 第十七节补充"打包命令固化"：
  - 打包命令必须严格从 `安装包/README.md` 标准模板复制粘贴，不得手敲删参数
  - 如需新增/修改参数，先更新模板再复制执行
