## 2. 各框架工具层深度分析

### 2.1 Claude Code
工具通过 `buildTool()` 工厂构造，每个工具有 `inputSchema`（Zod，双重用途：生成 JSON Schema 给 API + 运行时 `safeParse` 校验）、`call()`、`isConcurrencySafe(input)`（输入相关并发判定，`ls -la` 安全 / `rm -rf` 不安全）、`checkPermissions()`、`validateInput()`。fail-closed 默认：新工具默认串行、视为写操作、跳过自动安全分类器。14 步执行管线 `checkPermissionsAndCallTool()`：4 步校验 → 2 步准备 → 3 步权限（PreToolUse hooks → 权限裁决 → 拒绝处理）→ 5 步执行（含 PostToolUse hooks 与错误处理）。7 级权限模式：`default/acceptEdits/plan/dontAsk/bypassPermissions/auto/bubble`。**结果预算化**：每个工具声明 `maxResultSizeChars`，超限落盘到 `~/.claude/tool-results/{hash}.txt` 并以预览替换，会话级聚合预算防止"千刀万剐"。**工具延迟加载**：`shouldDefer: true` 仅发送 name+description，需 `ToolSearch` 调用才加载完整 schema，节省初始 prompt。MCP 工具与内置工具通过 `assembleToolPool()` 合并，内置优先 + MCP 其后，保持 prompt-cache 断点。40+ 工具分 5 类：File（Read/Edit/Write/NotebookEdit）、Search（Glob/Grep）、Execution（Bash/PowerShell/Monitor）、Web（WebSearch/WebFetch）、Code Intelligence（LSP）。

### 2.2 OpenAI Codex CLI
Rust 实现，`ToolHandler` trait 定义 `kind()/matches_kind()/is_mutating()/handle()`，`ShellHandler` 处理 `Function` 与 `LocalShell` 两种 payload。`ToolOrchestrator` 协调"审批 → 沙箱 → 执行"三段。`ShellRequest` 携带 command/cwd/timeout/env/proxy/sandbox 权限/justification。**沙箱分层**：Unix 用 landlock/seccomp，Windows 用 restricted token + 文件系统 ACL carving（read/write roots + deny paths）；`SandboxType` 由 `SandboxManager::transform` 根据 fs/network 策略决定。**审批缓存**：`ApprovalKey` 哈希（canonical command + cwd + permissions），同会话内相同命令不再追问。**优雅提权**：沙箱拒绝时可重试为非沙箱并提示用户。`unified_exec.rs` 用 PTY 持久 shell 会话，支持 `ZshFork` 模式。输出 capped 至 `EXEC_OUTPUT_MAX_BYTES`，`is_likely_sandbox_denied` 启发式检测沙箱失败。MCP 通过 `mcp_connection_manager` 路由非内置函数调用。

### 2.3 OpenAI Agents SDK
五类工具：Hosted OpenAI 工具（WebSearch/FileSearch/CodeInterpreter/ImageGen/HostedMCP/ToolSearch）、本地运行时工具（Computer/ApplyPatch/Shell/LocalShell）、Function calling（`@function_tool`）、Agents as tools（`agent.as_tool`）、Experimental（codex_tool）。`@function_tool` 装饰器从函数签名 + docstring（Google/Sphinx/NumPy 自动识别）通过 griffe + pydantic 自动生成 JSON Schema。关键参数：`name_override/description_override/docstring_style/use_docstring_info`、`strict_mode`（默认 True，严格 JSON schema）、`is_enabled`（动态启停）、`needs_approval`（bool 或 callable，人在回路）、`failure_error_function`（自定义错误回喂）、`timeout/timeout_behavior/timeout_error_function`、`defer_loading`（配 `ToolSearchTool` 用）、`tool_input_guardrails/tool_output_guardrails`、`custom_data_extractor`。`RunContextWrapper` 作为可选首参注入应用上下文（不发给模型）。可返回 `ToolOutputText/Image/FileContent` 多模态结构化输出。

### 2.4 Cursor
内置 12 个工具：Shell、Read、Write、StrReplace（局部编辑）、Grep（ripgrep）、Glob、SemanticSearch、Browser、Task（子 agent）、WebSearch、WebFetch、MCP。**多模型分工**：Main Agent LLM（推理+tool call）+ Embedding LLM（索引）+ Re-ranking LLM（查询重排）+ Apply Model（cheap 代码专用，把语义 diff 写成实际文件）+ Summarization LLM（历史压缩）。**沙箱默认**：Shell 写权限限 workspace、网络限已知包管理器、部分 syscall 受限；AI 可请求 `full_network/all` 提权但需用户审批。MCP 配置在 `~/.cursor/mcp.json`，每个 MCP server 工具缓存为 JSON schema 注入上下文。**状态less 循环**：每次请求都重建上下文（系统 prompt + 注入 context + 历史 + tool 结果），用 prompt caching 缓解成本。

### 2.5 Aider
无沙箱、无传统 function calling。`Coder` 工厂按 `edit_format` 实例化子类（search-replace/diff/whole-file/udiff/udiff-simple）。两类文件集：`abs_fnames`（可编辑）+ `abs_read_only_fnames`（只读上下文），自动检测文件提及 + glob 展开 + 编码校验 + `.aiderignore` + `.gitignore`。`Commands` 类实现 slash 命令：`/add`、`/drop`、`/read-only`、`/run`（`!` 前缀执行 shell）、`/test`（非零退出码才入 chat）、`/git`（输出不入 chat）、`/commit`（git attribution）。`SwitchCoder` 异常机制切换模式时重建 Coder。通过 litellm 支持任意 LLM（含本地 Ollama）。

### 2.6 Devin
Brain（stateless 云端推理）+ Devbox（Firecracker microVM 执行）双层架构。Devbox 三件套：shell、browser、code editor，Brain 通过递归 agentic loop（Plan-Act-Observe-Correct）驱动。**Firecracker 硬件级隔离**强于 Docker（不共享 host kernel），每个 session 独立 microVM，VM 跨 session 持久（保留装好的库）。CLI 4 级权限：Normal/Accept Edits/Bypass/Autonomous（配 `--sandbox`），autonomous 模式 shell 自动批准因沙箱已约束。作用域可动态扩展沙箱。网络按域名 allow/deny list 过滤。企业部署支持 Customer Dedicated VPC + AWS PrivateLink。

### 2.7 GitHub Copilot Agent Mode
内置工具 `read_file/edit_file/run_in_terminal` 等 + system prompt 指示 LLM 持续迭代到完成态。custom agent 通过 `.agent.md`/`.md` + YAML frontmatter 定义（name/description/tools/model/mcp-servers），`tools` 字段控制可用工具（`["*"]` 全开 / 指定 / `[]` 全关），工具别名 `execute/shell/read/edit/search/agent/web/todo`。预置 MCP server：`github`（只读）+ `playwright`（浏览器自动化）。VS Code 三类工具：built-in / MCP / extension，工具 picker 启停，3 级审批 Default/Bypass/Autopilot，单请求最多 128 工具。工具集（`.jsonc`）可组合并用 `#` 引用。Copilot CLI 内置子 agent：explore/task/general-purpose/code-review/research/rubber-duck。

### 2.8 LangChain
`@tool` 装饰器：函数 docstring → description，类型注解 → input schema。`@tool("name")` 改名，`@tool(args_schema=MyBaseModel)` 用 pydantic 定义复杂 schema，`@tool(return_direct=True)` 短路 agent loop。`StructuredTool.from_function` 程序化创建（含 `func` + `coroutine` 双实现）。`InjectedToolArg/InjectedToolCallId` 注解隐藏参数（不进 LLM schema，运行时注入），`ToolRuntime` 直接注入 state/context/store/stream_writer。返回类型：string / dict / multimodal / `Command`（状态更新）。错误处理：返回错误字符串而非抛异常，`handle_tool_error=True`。**注意**：非 JSON 可序列化的自定义类作为 tool arg 会触发 `PydanticInvalidForJsonSchema`（issue #34371），需用 pydantic BaseModel 包装。

### 2.9 LangGraph
`ToolNode` 预置节点处理并行执行 + 错误处理 + 状态注入。支持三种输入：graph state 含 `messages` key / message 列表 / 直接 tool calls。`handle_tool_errors` 可配置错误策略。`ToolRuntime[Context, State]` 注入工具访问 state/context/store/stream_writer/config/tool_call_id。普通工具返回 `ToolMessage`，命令工具返回 `Command`（更新 state / 触发导航 / 发消息）。三层 context：static runtime context（invoke 传入）/ dynamic state（运行时变）/ store（跨会话）。MCP 通过 adapter 挂载。

### 2.10 AutoGPT (Forge)
组件化架构取代旧 plugin 系统。`AgentComponent` 基类，组件实现 protocol：`DirectiveProvider/CommandProvider/MessageProvider/AfterParse/ExecutionFailure/AfterExecute`。命令用 `@command` 装饰 + JSON Schema 参数。内置组件 11 个：System（`finish` + API budget）、UserInteraction（`ask_user`）、FileManager（`read_file/write_file/list_folder`，支持 local/GCS/S3）、CodeExecutor（`execute_shell/execute_shell_popen/execute_python_code/execute_python_file`，Docker）、ActionHistory、GitOperations、ImageGenerator、WebSearch（DuckDuckGo/Google）、WebSelenium、Context、Watchdog（检测循环切 smart mode）。配置用 pydantic `ConfigurableComponent[BM]`。**Forge 已 legacy**，active 开发转向 AutoGPT Platform（可视化 graph builder，Polyform Shield 协议）。

### 2.11 CrewAI
`BaseTool` 子类（带状态/校验/env_vars/sync+async）或 `@tool("Name")` 装饰器（简单工具）。`args_schema: Type[BaseModel]` 用 pydantic 强校验。`_run()` 同步 / `_arun()` 异步。`cache_function` 自定义缓存策略。`env_vars` 声明所需环境变量。`CrewStructuredTool` 是内部表示，赋给 agent 时自动转换。设计原则：标准化接口、可扩展、模块化、lazy 依赖、一致文档。发布工具命名规范：包 `crewai-` 前缀、模块下划线、类 PascalCase 结尾 `Tool`。

### 2.12 Microsoft AutoGen
`FunctionTool` 类包装 Python 函数，类型注解 + description 自动生成 JSON schema。`ToolUseAgent` 持 `ChatCompletionClient` + Tool 列表，收到用户消息后请求 tool calls，`asyncio.gather` 并行执行所有 tool calls，结果包成 `FunctionExecutionResultMessage` 回喂模型反思。`ToolAgent` 专用执行 agent，抛 `ToolNotFoundException/InvalidToolArgumentsException/ToolExecutionException`。内置工具：`PythonCodeExecutionTool`（Docker）、`LocalSearchTool/GlobalSearchTool`（GraphRAG）、`mcp_server_tools`、`HttpTool`、`LangChainToolAdapter`。`Workbench` 管理共享资源/状态的工具集，支持动态可用性 + 生命周期（start/stop/reset）+ 状态持久化（save_state/load_state）。旧 0.2 API：`register_for_llm`（caller）+ `register_for_execution`（executor）+ `register_function`（双注册）。

### 2.13 Smolagents
核心创新：**CodeAgent**——LLM 写 Python 代码片段调用工具，而非 JSON tool call。支持函数嵌套/循环/条件，benchmark 显示比 JSON 方式少 30% 步骤且性能更高。`ToolCallingAgent` 是传统 JSON blob 方式（用 LLM 原生 tool calling）。`Tool` 基类 + `@tool` 装饰器 + 类属性 `name/description/inputs/output_type`。工具来源：`Tool.from_hub/from_langchain/from_space`、`ToolCollection.from_mcp`（Stdio/Streamable HTTP/SSE）。执行器：LocalPythonExecutor（**非沙箱**）/ Docker / E2B / Modal / Blaxel。ReAct loop：SystemPromptStep + TaskStep → while 循环 write_memory_to_messages → 模型 → 解析 → 执行 → ActionStep → 可选 PlanningStep → `final_answer`。模型无关：InferenceClientModel/LiteLLMModel/OpenAIModel/TransformersModel/AzureOpenAIModel/AmazonBedrockModel。

### 2.14 Pydantic AI
注册方式：`@agent.tool`（需 `RunContext`）/ `@agent.tool_plain`（纯函数）/ `tools=[...]` 构造参数 / `toolsets=[...]`。griffe 解析 docstring（Google/NumPy/Sphinx）。**Toolset 抽象**是核心：`AbstractToolset` 含 `get_tools/call_tool/get_instructions/for_run/for_run_step/__aenter__/__aexit__`。13+ 内置 toolset 类型：`FunctionToolset/CombinedToolset/FilteredToolset/PrefixedToolset/RenamedToolset/PreparedToolset/ApprovalRequiredToolset/DeferredLoadingToolset/IncludeReturnSchemasToolset/SetMetadataToolset/WrapperToolset/ExternalToolset/MCPToolset`，链式方法 `.filtered().prefixed().renamed().prepared().approval_required().defer_loading()`。执行流：`Agent.run() → UserPromptNode → ModelRequestNode → CallToolsNode`。高级：tool search 延迟加载（10+ 工具场景）、`prepare` 动态过滤、`ModelRetry` 显式重试、并行 tool call（`asyncio.create_task`，`sequential=True` 屏障）、`tool_choice`、per-tool timeout、自定义 args validator、人在回路审批。Capabilities 高阶 bundle（Thinking/WebSearch/WebFetch/Hooks/MCP 等）+ 生命周期 hooks。

### 2.15 LlamaIndex
`FunctionTool` 继承 `AsyncBaseTool`，`from_defaults()` 自动从函数 + docstring（Sphinx/Google/Javadoc）+ 类型注解生成 `fn_schema`（pydantic），过滤 `Context`/`self` 参数。同步/异步双向自动转换（`sync_to_async/async_to_sync`）。`ToolOutput` 含 `blocks`（ContentBlock 列表：Text/Image/Audio）+ `tool_name/raw_input/raw_output`。`FunctionAgent` 用 `llm.astream_chat_with_tools()` 拿 tool calls，scratchpad 存 workflow Context，支持 `return_direct` 短路。可 `to_langchain_tool/to_langchain_structured_tool` 互操作。

### 2.16 Semantic Kernel
`[KernelFunction]`（C#）/ `@kernel_function`（Python）标注函数，SDK 自动扫描生成 `KernelParameterMetadata` + JSON schema。auto function calling 6 步循环：序列化函数 → 发模型 → 模型返回 chat 或 function call → 提取 name/params → invoke → 结果回喂 → 重复直到 chat 或 max iterations。支持并行 function call。保留参数名 `kernel/service/execution_settings/arguments` 自动注入不广告给 LLM。三种 plugin 导入：Native code / OpenAPI spec / MCP Server。建议：≤20 工具（最好 ≤10）、snake_case 命名、最少参数、优先 primitive/enum、return type 简单、敏感大数据用 local state 不传 LLM。

### 2.17 MetaGPT
`Action` 基类（`name/i_context/prefix/desc/node/llm_name_or_type` + `run()/_aask()/set_prefix()`）+ `Role` 基类 Observe-Think-Act loop。三种 react_mode：REACT（LLM 选）/ BY_ORDER（顺序）/ PLAN_AND_ACT（先规划）。标准 SOP 5 角色 ProductManager→Architect→ProjectManager→Engineer→QaEngineer，对应 Action：WritePRD/WriteDesign/WriteTasks/WriteCode/WriteCodeReview/SummarizeCode/WriteTest/RunCode/DebugError。`ActionNode` 树状 prompt schema + 动态 pydantic 模型生成 + fill 策略（simple/complex/code_fill/xml_fill/single_fill）+ review/revision（auto/human）。Function calling 通过 `aask_code` + `GENERAL_FUNCTION_SCHEMA`（execute(language, code)），4 种响应解析回退（标准 tool call → JSON decode err 回退 regex → markdown code block → plain text）。RoleZero 用 `tool_execution_map` + `@register_tool` 动态调度 action。

### 2.18 Jan
桌面端 Tauri（Rust）+ TypeScript，本地 llama.cpp，OpenAI 兼容 server @ 1337。Jan Server（Go）微服务架构：Kong Gateway + LLM API/Response API/Media API/MCP Tools(8091)/Memory Tools(8090)/Realtime API。MCP Tools Service 代理 Serper/SearXNG/scraping/file search/SandboxFusion。Response API 迭代 `tools/list`+`tools/call`，受 `RESPONSE_MAX_TOOL_DEPTH` 与 `TOOL_EXECUTION_TIMEOUT` 约束。新 `/v1/orchestrations` 端点（PR #7800）让无头 API 客户端也能跑工具循环。桌面启用 MCP 需开 Advanced 实验功能 + Settings > MCP Servers + 选支持 tool 的模型。

### 2.19 LM Studio
OpenAI 兼容 `/v1/chat/completions` tools 数组。原生 tool use（hammer 徽章）：Qwen2.5/Llama-3.1/3.2/Mistral/Ministral。其他模型走 fallback `[TOOL_REQUEST]...[END_TOOL_REQUEST]` 自定义 system prompt。MCP 0.3.17+ 引入，0.4.0+ API 支持：ephemeral MCP（per-request `integrations` 字段）+ pre-configured `~/.lmstudio/mcp.json`（`integrations: ["mcp/playwright"]` 引用）。工具调用确认弹窗 + always allow + `allowed_tools` 限制 + 加速 prompt 处理。Python/TypeScript SDK `.act()` 自动多轮 tool calling，异常自动回喂 LLM。推荐 Qwen2.5-7B-Instruct。

### 2.20 Ollama
v0.3.0 引入 tool calling。Go template 驱动：用 `text/template` 格式化 tool 定义进 prompt，parser 用状态机从流式输出搜模型特定 tag（`[TOOL_CALLS]` Qwen / 不同格式 DeepSeek/Mistral）。Python SDK 可直接传函数，自动从 docstring + type hints 解析为 schema。流程：messages + tools → 模型返回 `tool_calls` → client 执行 → 结果作 `tool` role 消息追加 → 再调模型。支持单 shot / 并行 / 多轮 agent loop / 流式（tool_calls 在末尾 delta）。OpenAI 兼容 `/v1/chat/completions` + Anthropic 兼容（`tool_use/tool_result` block + Web Search Loop）。推荐 llama3.1:8b（8/10）/ qwen2.5:14b（9/10）。最佳实践：description < 200 字、6-8 工具上限、`tool_choice` 强制、`max_turns` 防失控、try/except 返回结构化错误 JSON。

### 2.21 GPT4All
C++20 + Qt6/QML + llama.cpp + ggml，GPU via Vulkan/Metal/CUDA。`LLModel` 抽象基类 + `ChatLLM`（管理对话状态 + 在独立 Qt 线程解析 tool call）+ `LocalDocs`（SQLite 嵌入式向量 + BM25 混合搜索，全离线 RAG）+ `Server`（QHttpServer OpenAI 兼容端点）。原生 C++ Jinja2 兼容 chat template 引擎（无需 Python）。Python binding `pip install gpt4all` + OpenAI 兼容本地 server（Docker）+ Code Interpreter（设备端 JS 沙箱）。Tool calling 在 ChatLLM 层已有 parser，但 chat UI 全功能 tool 支持仍在社区讨论（#3599），倾向 MCP。支持模型：Llama 3.1 / Mistral 7B v0.3+ / Hermes / Qwen2.5。

### 2.22 AnythingLLM
`@modelcontextprotocol/sdk` + `StdioServerTransport`，三层架构：MCP Protocol Layer → Tool Layer → Client Layer（`AnythingLLMClient` HTTP）。39+ 工具双路路由：11 core tools（inline `src/index.js`，workspace/chat/document/system settings）+ 28 additional tools（`src/additional-tools.js`，user/API key/embeddings/agents/LLM/vector config/search）。`CallToolRequestSchema` 两阶段：解析 → 校验 → 路由 switch → 执行 → 包装 → 错误处理。错误统一 `{content:[{type:'text',text:...}],isError:true}`，catch-all 保证 server 不崩。PR #3547（2025-03）为原生 AnythingLLM 带来完整 MCP：`default mcp_config.json` + `anythingllm.autoStart` 开关 + 动态重载 + Agent Skills 管理 MCP + 内置 agent 框架调用运行中 MCP + 子进程管理 + `npx`/`uv`/`uvx` Docker 支持。5 个 agent 管理 tool：list/create/update/delete/invoke，最小权限（tools 数组白名单）+ 持久 registry + workspace 隔离。

### 2.23 OpenWebUI
两类扩展层：In-Process Python（Tools/Pipe/Filter/Action/Skills）+ External HTTP（Native MCP HTTP / MCPO 代理 stdio→HTTP / OpenAPI Servers）。Tools 脚本持有 server 端密钥（user 和 model 都读不到）。Native Mode（v0.6.31+，唯一支持模式）用结构化 tool call 不破坏 KV cache，自动注入系统工具：`search_web/fetch_url/list_knowledge/query_knowledge_files/grep_knowledge_files/view_file/kb_exec/search_memories/add_memory/delete_memory/list_memories/search_notes/view_note/write_note/search_chats/view_chat/search_channels/search_channel_messages/create_tasks/update_task/create_automation/list_automations/toggle_automation/search_calendar_events/create_calendar_event/execute_code/generate_image/edit_image/get_current_timestamp/calculate_timestamp`。MCP 原生仅支持 Streamable HTTP（多租户匹配 / 进程管理 / 并发冲突 / 容器限制 / 安全），stdio 通过 mcpo 桥接（`uvx mcpo --port 8000 -- your_mcp_server`，自动生成 FastAPI + Swagger）。OAuth 2.1 + PKCE + DCR + 每用户独立认证。9400+ 公开 MCP server（2026-04，YoY 7.8x）。

### 2.24 MCP（协议本身）
client-host-server 架构 + JSON-RPC 2.0。Host（Claude Desktop/VS Code）管多 client 实例 + 安全策略 + 用户授权；Client 1:1 stateful session 与 Server；Server 暴露 resources/tools/prompts。设计原则：server 极易构建 / 高可组合 / 不能读整个对话或窥其他 server / 通过 capability negotiation 渐进加功能。两层：Data Layer（JSON-RPC lifecycle + tools/resources/prompts + sampling/elicitation/logging）+ Transport Layer（stdio 本地 / Streamable HTTP 远程，OAuth 推荐）。Tool 定义：`name`(1-128 字符 `[A-Za-z0-9_-\.]`)/`title`/`description`/`icons`/`inputSchema`(JSON Schema 2020-12)/`outputSchema`/`annotations`。结果：非结构化（`content` 数组 text/image/audio/resource_link/embedded_resource）或结构化（`structuredContent` 符合 `outputSchema`）。两类错误：Protocol Error（JSON-RPC error）/ Tool Execution Error（`isError:true`，LLM 可据以行动）。多轮请求：`InputRequiredResult` + `inputRequests` + `requestState`。`notifications/tools/list_changed` + 分页 cursor + 缓存 `ttlMs/cacheScope`。

## 3. 模式归纳

### 3.1 工具注册：四大主流范式
1. **装饰器 + 函数签名协议**：LangChain `@tool`、CrewAI `@tool`、Smolagents `@tool`、AutoGen `FunctionTool`、Pydantic AI `@agent.tool`、LlamaIndex `FunctionTool.from_defaults`、Q-agent `@tool`——签名即协议，inspect.signature + pydantic 自动 schema 是事实标准。
2. **基类继承 + pydantic args_schema**：CrewAI `BaseTool`、Smolagents `Tool`、AutoGPT `AgentComponent`——适合有状态/复杂校验工具。
3. **工厂函数 + Zod/JSON Schema**：Claude Code `buildTool()`、OpenAI Agents SDK `FunctionTool(...)` 直构、MCP server `inputSchema`——schema 即单一真相源。
4. **方法标注 + 自动发现**：Semantic Kernel `@kernel_function`、MetaGPT `@register_tool`、AutoGPT `@command`——SDK 扫描注册。

跨框架共识：**docstring → description + 类型注解 → schema** 自动生成是标配；pydantic（Python 生态）或 Zod（TS 生态）是 schema 首选；description 质量 > 命名（模型按 description 选工具）。

### 3.2 安全：四档光谱
- **无沙箱派**：Aider（`.aiderignore` + 只读集 + git）、MetaGPT、LangChain/LangGraph（由运行时负责）、Q-agent（永不沙箱 + 黑名单 + 路径保护）——靠输入校验 + 文件分级 + 用户审批。
- **轻约束派**：Claude Code（7 级权限 + fail-closed 默认 + hooks，无 OS 沙箱但权限分层精细）、Cursor（workspace 写限 + 包管理器网络白名单）、LM Studio（确认弹窗 + allowed_tools）。
- **OS 沙箱派**：OpenAI Codex CLI（landlock/seccomp + Windows restricted token + ACL carving）、Devin（Firecracker microVM）——硬件级/内核级隔离。
- **容器沙箱派**：AutoGPT CodeExecutor（Docker）、AutoGen `PythonCodeExecutionTool`（Docker）、Smolagents（E2B/Docker/Modal/Blaxel）——代码执行走容器。

危险命令拦截：Claude Code PreToolUse hooks + 安全分类器、Codex `is_likely_sandbox_denied` 启发式、Q-agent safety 黑名单——黑名单+白名单混合是主流。

### 3.3 工具调用接口：三格式鼎立
1. **OpenAI function calling**（`tools` 数组 + `tool_calls` 响应 + `tool` role 回喂）：Codex、Cursor、Agents SDK、CrewAI、AutoGen、Pydantic AI、Semantic Kernel、LangChain、LM Studio、Ollama、Jan——事实标准。
2. **Anthropic tool_use block**：Claude Code、Ollama Anthropic 兼容层。
3. **Prompt 解析**（自定义标记）：LM Studio fallback `[TOOL_REQUEST]`、OpenWebUI legacy mode（已废弃）、Q-agent dispatcher 解析 ` ```json ... ``` ` 块、MetaGPT markdown code block 回退、Smolagents CodeAgent（写 Python 代码）。

**KV cache 友好性**是当代分水岭：OpenWebUI 专为此废弃 legacy prompt injection 模式转向 Native Mode，因为 prompt 注入每轮破坏 cache。Q-agent dispatcher 的 json block 解析属 prompt 注入路径，需评估 cache 影响。

### 3.4 结果回喂：四策略
- **截断阈值**：Codex `EXEC_OUTPUT_MAX_BYTES`、Claude Code `maxResultSizeChars`、Q-agent ContextManager L1 落盘。
- **落盘 + 占位符**：Claude Code 落盘 `~/.claude/tool-results/{hash}.txt` + 预览替换（最精细）、Q-agent L1 工具结果落盘。
- **会话级预算**：Claude Code 聚合 conversation budget 防"千刀万剐"——独有。
- **直接回喂**：大多数框架无特殊处理，依赖 LLM 上下文窗口 + 摘要压缩。

### 3.5 错误处理：三范式
- **结构化错误回喂 LLM**：Ollama 最佳实践（try/except 返回结构化 error JSON）、MCP `isError:true`（LLM 可行动）、AutoGen `ToolExecutionException`、LM Studio SDK（异常自动回喂）——主流。
- **重试机制**：Pydantic AI `ModelRetry` + per-tool retry limit、AutoGen 反思循环、Smolagents self-heal。
- **PermissionError 单独处理**：Claude Code denied handling 单独分支、Codex sandbox denied 启发式 + 提权重试、Devin 4 级权限模式分支——权限错误需区别于普通执行错误。

### 3.6 持久化
- **sqlite**：Q-agent（sessions/messages/compaction_records）、GPT4All LocalDocs（嵌入式向量 + BM25）、OpenWebUI（MCP tool schema 入库）。
- **JSONL/文件**：Claude Code tool-results 落盘、Cursor SQLite 缓存 tool schema。
- **不入库**：大多数框架工具调用记录随对话历史走（LangChain memory / AutoGen history / Smolagents ActionStep）。

### 3.7 并发/异步
- **asyncio.gather 并行**：AutoGen、Pydantic AI（`asyncio.create_task` + `sequential=True` 屏障）、OpenAI Agents SDK、LangGraph ToolNode 并行执行。
- **QThread**：Q-agent ChatWorker、GPT4All ChatLLM 独立 Qt 线程。
- **子进程 + 超时**：Codex `spawn_child_async` + timeout/cancellation、Aider `run_cmd()`、Devin Devbox shell。
- **PTY 持久 shell**：Codex `unified_exec` PTY backed、Claude Code Bash。

### 3.8 MCP 支持：三档
- **原生首等公民**：Claude Code、Cursor、Codex CLI、OpenAI Agents SDK（HostedMCPTool）、LM Studio、Jan、OpenWebUI、AnythingLLM、VS Code Copilot——`mcp.json` 配置 + tool schema 自动注入 + tool calling 路由。
- **Adapter 挂载**：LangChain `langchain-mcp-adapters`、LangGraph、LlamaIndex、Smolagents `ToolCollection.from_mcp`、Pydantic AI `MCPToolset`、Semantic Kernel MCP plugin、AutoGen `mcp_server_tools`、CrewAI crewai-tools MCP——通过 adapter 把 MCP server 转为本框架 tool。
- **不支持/讨论中**：Aider、MetaGPT、GPT4All（社区讨论中）。

MCP 已成为跨框架最大公约数：9400+ 公开 server，SDK 覆盖 TS/Python/Java/Kotlin/C#/Swift，月下载 9700 万。

### 3.9 本地 LLM 适配
- **原生本地**：Ollama（Go template 驱动）、LM Studio（OpenAI 兼容 + fallback prompt）、GPT4All（C++ ChatLLM parser）、Jan（vLLM/llama.cpp）、OpenWebUI（任意 provider）。
- **通过 litellm/SDK**：LangChain、CrewAI、AutoGen、Pydantic AI、LlamaIndex、Smolagents（LiteLLMModel/TransformersModel）——本地 LLM 通过 OpenAI 兼容端点接入。
- **本地 tool calling 路径**：原生 function calling（Qwen2.5/Llama 3.1/Mistral 等支持模型）+ prompt 解析 fallback（不支持模型）——双轨制是本地 LLM 工具调用标配。

## 4. 对 Q-agent 的启示

### 4.1 Q-agent 当前设计定位
Q-agent v0.0.18 当前工具层设计：
- `@tool(name, desc)` 装饰器 + 函数签名协议（参数名+类型+返回类型）+ `inspect.signature` 注册校验——**完全契合主流装饰器 + 签名协议范式**，与 LangChain/CrewAI/Smolagents/Pydantic AI 同源。
- safety 黑名单 + 路径保护（G:\agent 禁区）+ 永不沙箱——**属"无沙箱派"**，与 Aider/MetaGPT/LangChain 同阵营，但比 Aider 多了黑名单 + 路径保护。
- dispatcher 从 LLM 输出解析 ` ```json ... ``` ` 块（支持 OpenAI function 格式 + 自动生成 call_xxx id）——**属 prompt 解析路径**，与 LM Studio fallback、OpenWebUI legacy、MetaGPT markdown 回退同列。
- ContextManager 4 级压缩（L1 工具结果落盘 / L2 旧消息截断 / L3 异步真摘要 / L4 硬溢出）——**比多数框架精细**，Claude Code 的 `maxResultSizeChars` + 落盘 + 会话预算是唯一更精细的对照。
- sqlite3 持久化（sessions/messages/compaction_records）——与 GPT4All/OpenWebUI 同档，比 LangChain/AutoGen 默认不入库更扎实。

**优势**：
1. 装饰器 + 签名协议是跨框架事实标准，Q-agent 选型正确，未来生态互操作（如转 LangChain tool / MCP tool）成本低。
2. 4 级压缩 + L1 工具结果落盘 + sqlite 持久化是"工具结果治理"的高配置路径，已超过 LangChain/AutoGen/CrewAI 默认水平，接近 Claude Code。
3. 永不沙箱 + 黑名单 + 路径保护符合"桌面端本地工具"定位（用户已在自己的机器上，沙箱价值低于云端 Devin/Codex），与 Aider 同哲学。
4. dispatcher 同时支持 OpenAI function 格式 + 自动生成 call_id——兼容性好，未来切原生 function calling 时迁移成本低。

**劣势 / 风险**：
1. **prompt 解析路径破坏 KV cache**：OpenWebUI 专为此废弃 legacy 模式。Q-agent 当前 dispatcher 解析 json block，对支持原生 tool calling 的本地模型（Qwen2.5/Llama 3.1）是退步。建议 v0.0.19 优先支持 Ollama 原生 `tool_calls`，prompt 解析作为不支持模型的 fallback。
2. **无并发判定**：Claude Code `isConcurrencySafe(input)` 输入相关并发判定 + Codex 审批缓存是中高端配置，Q-agent 当前未涉及——v0.0.19 若多工具并行需补。
3. **无错误回喂 LLM 范式**：Ollama/MCP/AutoGen 共识是 try/except 返回结构化 error JSON 让 LLM 自愈，Q-agent 需明确此策略。
4. **无 PermissionError 单独分支**：Claude Code/Codex/Devin 都把权限错误与执行错误分开，Q-agent safety 拦截后如何回喂 LLM 需设计。
5. **无 MCP 支持**：MCP 是跨框架最大公约数，Q-agent 当前未涉及——长期看是必选项。
6. **无工具延迟加载**：Claude Code `shouldDefer` + ToolSearch、Pydantic AI DeferredLoadingToolset、OpenAI Agents SDK defer_loading——工具数超 10 个时是刚需，Q-agent 4 类候选暂不急需但需预留接口。

### 4.2 v0.0.19 工具层实化可借鉴的设计点
1. **Ollama 原生 tool calling 双轨制**（最高优先级，参考 Ollama + LM Studio）：对支持模型走 `tool_calls` 原生路径（KV cache 友好），不支持模型走 dispatcher prompt 解析 fallback。这是当前 dispatcher 已有的能力扩展，不是重写。
2. **结构化错误回喂**（参考 Ollama 最佳实践 + MCP `isError:true` + AutoGen ToolExecutionException）：所有工具 try/except，PermissionError / 执行错误 / 超时三类分别返回结构化 JSON 让 LLM 区分行动。
3. **工具结果预算 + 落盘**（参考 Claude Code `maxResultSizeChars`）：Q-agent ContextManager L1 已有落盘，建议补"会话级聚合预算"——单工具截断 + 总预算双重控制，防"千刀万剐"。
4. **并发安全标识**（参考 Claude Code `isConcurrencySafe` + Pydantic AI `sequential=True`）：`@tool` 装饰器可选参数 `concurrency_safe: bool` 或 `sequential: bool`，dispatcher 调度时据此外加锁或并行。
5. **保留 call_id 关联**（参考 Codex + OpenAI function calling）：Q-agent dispatcher 已自动生成 call_xxx id，需确认 tool_result 回喂时 id 关联正确，便于 LLM 多工具并行结果对齐。
6. **危险命令黑名单 + 路径保护已到位**：v0.0.19 在此基础上补 PermissionError 回喂格式即可，无需引入沙箱（符合永不沙箱约束）。

### 4.3 v0.0.19 工具范围（4 类候选）优先级建议
按"用户价值 × 实现成本 × 安全风险"排序：

| 优先级 | 工具类 | 理由 |
|--------|--------|------|
| P0 | **文件操作**（read_file/write_file/list_dir/move/search） | 最高频，Aider/Claude Code/Cursor 都以文件为核，无网络依赖，安全风险最低（路径保护已就位），search 可复用 ripgrep 思路 |
| P1 | **shell**（execute_shell + safety） | Claude Code Bash / Codex Shell / Cursor Shell 都列为骨干，safety 黑名单已有基础，需补超时 + 输出截断 + PermissionError 回喂 |
| P2 | **网络**（http_get/web_fetch） | 价值高但本地 LLM 场景需谨慎（用户可能离线），实现简单（标准库 urllib），安全风险中等（无沙箱需 URL 白名单或仅 HTTPS） |
| P3 | **代码执行**（python_eval 子进程） | 价值高但**安全风险最高**（无沙箱执行任意代码），Devin/AutoGPT/AutoGen/Smolagents 都强制 Docker/E2B 沙箱，Q-agent 永不沙箱约束下需慎重——建议 v0.0.19 暂缓或仅限"只读 eval"（不 import os/subprocess、不写文件） |

建议 v0.0.19 范围：P0 + P1 落地，P2 视用户需求增量，P3 暂缓或做"受限 eval"子集。

### 4.4 未来 MCP 适配路径建议
MCP 是跨框架最大公约数（9400+ server，6 语言 SDK，月下载 9700 万），Q-agent 长期必接。建议路径：

1. **短期（v0.0.19-v0.0.20）**：先把 4 类内置工具按 `@tool` 装饰器协议落地，验证 dispatcher + ContextManager + sqlite 全链路。MCP 不进 v0.0.19。
2. **中期（v0.0.21+）**：实现 MCP **client** 端（Q-agent 作为 host），通过 stdio 连本地 MCP server（参考 Claude Code `assembleToolPool()` 模式：内置 tool 优先 + MCP tool 其后）。Streamable HTTP 暂缓（多租户/认证复杂，桌面单用户场景 stdio 足够）。
3. **长期**：评估是否实现 MCP **server** 端（把 Q-agent 工具暴露给其他 AI 工具调用），这取决于 Q-agent 是否想成为生态公民。
4. **架构预留**：v0.0.19 的 `@tool` 注册器应设计为"可注入外部 tool 源"——未来 MCP tool 转换为本框架 tool 时只需实现 `name/description/input_schema/call()` 四元组，不需改 dispatcher。

技术选型建议：直接用官方 `mcp` Python SDK（modelcontextprotocol/python-sdk），避免自造协议；stdio transport 优先，与 Q-agent 桌面单用户定位匹配。
