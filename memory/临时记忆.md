## 七、未解决问题

- github.com push 网络问题（v0.0.15 修订 push 失败错误 060，v0.0.16 待 push）
- .exe stdout 在外部进程调用时不输出（PyInstaller --onefile + --windowed GUI 应用 console handle 继承）——不影响用户体验
- 取消按钮未接 ChatWorker.stop()（接口已留 v0.0.8，待 v0.0.17+ 候选 1）
- QSettings 持久化完全空白（host 仍 hardcoded DEFAULT_HOST，模型选择未持久化，settings_page 控件无落盘）——需先出方案避免散弹式侵入三模块
- 历史消息首版全传（token 消耗大但简单），上下文裁剪留 v0.0.17+
- 设计规范 spacing/shadow 表是推测值
- CLOUD_PRESET 仍是硬编码占位，未接 OpenAI/Anthropic/Google API（cloud stubs 仅骨架）
- Ollama Cloud 判定依赖 remote_model/remote_host 字段，老版 Ollama 可能不返回，退化为 v0.0.6 行为（graceful，非问题）
- release_model 调用时如果模型正在 chat 生成中，Ollama 在当前生成完成后立即卸载（keep_alive=0 不强制中断当前生成）；验证步骤会检测到模型仍在 → 抛错提示"可能正在生成中，请稍后重试"
- 切换模型清空对话是硬规则，不保留历史；用户切回原模型需重新输入
- 模型名 hash 颜色可能碰撞（crc32 mod 8 同值），8 色调色板日常足够，碰撞概率 1/8 可接受
- toolbar"关于"按钮仍是活 UI 空壳（status_callback 占位），未来需要打开关于弹窗时可同模式扩展 about_requested 信号
- LoadingDots 固定 dot_size=8 不随 DPI 缩放（高 DPI 屏可能偏小）
- LoadingDots 流动彩虹色相独立于模型色，与"模型名 hash 色"色彩系统并行（设计取舍，非冲突）
- **Ollama 进程级 CUDA context 不立即归还 OS**：release_model /api/ps 验证通过 + nvidia-smi VRAM 已归还，但任务管理器看 ollama.exe 进程的"专用 GPU 内存"列可能仍显示几 GB 占用，这是 Ollama 已知行为非项目 bug，状态栏文案已明确告知用户
- **硬件监控无 NVIDIA 显卡时 GPU/VRAM/GPU°C 永远 N/A**：pynvml 容错降级，AMD/Intel GPU 不支持（需引入额外依赖，等用户提需求）
- **硬件监控 CPU°C 在 Windows 永远 N/A**：psutil 不支持 sensors_temperatures（Linux/macOS 支持），避免引入 wmi/LibreHardwareMonitor 等第三方库，符合"零第三方依赖起步"原则
- **硬件监控采样间隔固定 1s 不可配置**：未来可在设置中调 1s/2s/5s
- **硬件监控历史窗口固定 60s 不可配置**：未来可调 60s/120s/300s
- **硬件监控不写盘持久化**：关闭重启后历史清空
- **硬件监控窗口固定 620×520 不可调整大小**：未来可加可拖拽 resize
- **硬件监控窗口不置顶**：未来可加 always-on-top 选项
- **v0.0.12/v0.0.13 的教训已通过 ADR-027/028 + CLAUDE.md 第二十一节明文化**：贴纸式开发原则 + 打包命令固化，下次 AI 在规则层面被拦住，不靠判断力
- **v0.0.16 候选审查教训**：用户对"未解决问题很久了，是不是反复改同一块代码"有合理质疑——AI 实际审查代码后确认三个候选都是扩展公开接口非侵入式，与 v0.0.12 错误性质完全不同。审查报告可作为未来类似质疑的参考范式
